# Nova_Store ‚ÄĒ Admin Dashboard & System Management Portal: Enterprise Blueprint

> **Status:** Planning / Blueprint (gap analysis + phased implementation roadmap)
> **Audience:** Engineering / Product / Architecture
> **Stack (existing, do not change):** Express 5 (CommonJS) + Supabase (Postgres, RLS disabled on admin tables) + Redis (queue/cache/pub-sub) + React 18 (TanStack Query) admin SPA. The platform operates exclusively in **Nigerian Naira (NGN)** with **gross pricing (no tax computation)** and a **single-vendor** business model. ## 1. Advanced System Settings & Global Configuration

**Objective:** Replace the ad-hoc flat-KV settings with a structured, grouped, type-safe configuration platform that drives storefront rendering and shipping. **Constraint:** platform is **NGN-exclusive** (single currency, no FX/conversion) and uses **gross pricing** (no tax engine).

### 1.1 Single Currency ‚ÄĒ Nigerian Naira (NGN) ONLY
- **Scope decision:** **Remove all multi-currency support and internationalization logic.** The platform operates with **NGN as the single, immutable default currency** for every transaction, product price, and financial report. No currency selector, no exchange-rate table, no conversion service.
- **Sub-features (retained):** NGN symbol/format locale (`‚ā¶`, kobo precision), a consistent money-formatting helper `money.format(naira)` used across storefront + admin, and a single `settings` entry `currency.default = 'NGN'` (read-only constant, not user-editable).
- **Functional importance:** A single-vendor Nigerian retailer eliminates FX risk, reconciliation drift, and cross-currency pricing bugs. Removing multi-currency dramatically simplifies the pricing/checkout/ledger data model.
- **Nova_Store gap:** `StoreSettings.tsx` hardcodes a 4-option currency `<select>` (NGN/USD/EUR/GBP) on `stores.currency`. Must be reduced to a fixed NGN label (no select).
- **Proposed:** **No `currencies` / `exchange_rates` tables, no `currency.service.js`.** Constrain `stores.currency` to `DEFAULT 'NGN'` and remove the select UI. Add a shared `money.util.js` for NGN formatting only.

### 1.2 Locale & Time Zone (Operational Only ‚ÄĒ NO i18n)
- **Scope decision:** **Remove all internationalization (i18n) / multilingual logic.** No translatable catalog, no language switcher, no per-locale content.
- **Sub-features (retained, operational):** A single canonical **store timezone** (drives "today's orders", reporting, analytics) and **date/number display formats**. English is the only UI language.
- **Functional importance:** Operational dashboards are meaningless without a canonical timezone, but language localization adds no value for a single-market Nigerian store and is removed to cut complexity.
- **Proposed:** `settings` group `localization` `{store_timezone, date_format}` only (no `default_locale`/`enabled_locales`). Wire `store_timezone` into analytics views (`059_analytics_timezone_fix` already notes this need).

### 1.3 Tax Engine ‚ÄĒ REMOVED FROM SCOPE
- **Scope decision:** **Remove the Tax Engine and all automated tax calculation modules.** Pricing is handled as **gross NGN amounts**; there is no integrated tax computation or any tax-related settings/permissions.
- **Impact:** No `tax_rates` / `tax_classes` / `tax_zones` tables, no `tax.service.js`, no `tax:read`/`tax:write` permissions. Invoices (¬ß5.2) and financial reports render gross NGN totals only. If statutory VAT reporting is later required, it is added as a separate, explicitly-scoped compliance module ‚ÄĒ not woven into the pricing path.

### 1.4 Shipping Zone Logic
- **Sub-features:** Geographic zones, rate models (flat/weight-based/price-threshold/free-over-X), carrier rules, per-zone ETA, restricted items.
- **Functional importance:** Shipping cost directly drives cart conversion; misconfigured zones cause margin loss.
- **Proposed:** Extend `shipping_zones` (`026`) with `rate_strategy` JSON (all rates in **NGN**); admin UI in `features/shipping` (exists) to manage zones + rates. Keep current `shipping.routes.js`.

### 1.5 Global SEO / Metadata
- **Sub-features:** Store-level `meta_title/description/OG tags`, robots.txt, sitemap generation, structured-data (JSON-LD) toggle, canonical rules, social share cards.
- **Functional importance:** Organic discovery is the lowest-CAC channel at marketplace scale.
- **Proposed:** `settings` group `seo` + a `seo.service.js` that renders sitemap/robots and injects JSON-LD. Per-product SEO columns already exist (`meta_title` etc.) ‚ÄĒ reuse.

**Module endpoints:** `GET/PUT /admin/settings/:group`, `GET/POST /admin/shipping-zones`. (No currency or tax endpoints ‚ÄĒ NGN is fixed; tax is out of scope.)

## 2. Granular Role-Based Access Control & Profile Management

**Objective:** A unified, auditable permission system covering the 6 target roles with defense-in-depth security controls.

### 2.1 Unified Role Model (replace dual-system drift)
- **Target roles:** `SUPER_ADMIN`, `CATALOG_MANAGER`, `LOGISTICS_COORDINATOR`, `CUSTOMER_SUPPORT`, `FINANCE_AUDITOR`, `MARKETING_SPECIALIST`. Keep existing `STORE_OWNER` as alias of `SUPER_ADMIN` for backward compat.
- **Permission levels (verbs):** `VIEW`, `CREATE`, `EDIT`, `DELETE`, `APPROVE`. Model as permission keys `resource:verb` (e.g., `catalog:approve`, `crm:read`, `finance:read`, `marketing:write`, `logistics:write`). **Note:** No `tax:*` permissions (tax removed).
- **Functional importance:** Least-privilege containment limits blast radius of a compromised or rogue admin account ‚ÄĒ essential at scale with many operators.

### 2.2 Permission Catalog Expansion
Extend `permissions` seed with domains: `system`, `rbac`, `catalog`, `inventory`, `orders`, `shipping`, `logistics`, `marketing`, `crm`, `finance`, `analytics`, `audit`. Each with `*:read/write/create/delete/approve` as applicable. **Omit `tax` domain** (tax removed). Reuse existing keys (`product:create`, `order:write`, etc.) ‚ÄĒ do not break.

### 2.3 Security Protocols
- **2FA (TOTP/WebAuthn):** `admin_security` table (`two_factor_enabled`, `totp_secret` encrypted, `recovery_codes` hashed). Enrollment + verify flow in `auth.routes.js`/`session.routes.js`.
- **Session Management:** Active-session list, revoke-others, device fingerprinting (column already added in `023_catalog_audit_extensions`: `device_fingerprint`), idle + absolute timeout, concurrent-session cap.
- **IP Whitelisting:** `admin_ip_allowlist` table; middleware enforced for `SUPER_ADMIN`/sensitive roles; bypass for break-glass with audit.
- **Audit Logs:** Already strong (`audit_logs` hash chain). Add **non-repudiation verification UI** (recompute `record_hash` chain) and RBAC redaction (see prior plan `audit-redaction.middleware.js`).

**Module endpoints:** `GET/POST/PUT /admin/rbac/roles`, `POST /admin/rbac/roles/:id/permissions`, `POST /admin/security/2fa/enable`, `GET /admin/sessions`, `DELETE /admin/sessions/:id`, `GET/POST /admin/security/ip-allowlist`.

## 3. Catalog & Inventory Governance  ‚ėÖ PHASE 1 (detailed below)

**Objective:** Enterprise catalog modeling (variants as option matrices, category hierarchy), bulk operations, multi-location stock, and proactive alerting.

### 3.1 Multi-Variant Attributes (option matrix)
- **Current:** `product_variants.option_values` is freeform JSONB; no canonical option groups.
- **Proposed:** `product_options` (name: Size/Color), `product_option_values` (S/M/L, Red/Blue), `product_variant_options` (variant‚Üívalue map). Variants become the Cartesian product of selected options with per-variant SKU/price/stock. Enables correct filtering & PDP selection.
- **Importance:** Variant integrity is the #1 cause of oversell/return disputes in marketplaces.

### 3.2 Bulk Upload / Import (Excel `.xlsx` / `.xls`)
- **Scope decision:** Replace the JSON/CSV bulk upload with an **Excel-based ingestion system**. Import logic parses spreadsheet **columns** mapped to product attributes, inventory, and pricing.
- **Proposed:** `import_jobs` table (status, totals, error rows, `file_format` = `xlsx|xls`); an Excel parser service `bulk-import.service.js` built on a streaming XLSX reader (e.g., `exceljs`/`xlsx`) that maps **named worksheet columns ‚Üí entity fields**, validates cell types (numbers, dates, enums), and enqueues a Redis worker (reuse `jobs/`). Error rows are written to a downloadable **`.xlsx` reject sheet** with per-cell error annotations. Supports products, variants, inventory, categories worksheets.
- **Importance:** Merchants onboard 10k+ SKUs; spreadsheet column mapping is the familiar, low-error ingestion path for non-technical catalog managers. Streaming parse avoids memory blowups on large workbooks.

### 3.3 Stock Threshold Alerts
- **Proposed:** Move `low_stock_threshold` to per-product + per-warehouse; `stock_alert_rules` (threshold, channel, recipients). `inventory-alert.service.js` emits domain event on crossing ‚Üí notification (reuse `notification-router.service.js`).
- **Importance:** Prevents stockouts that directly cost GMV.

### 3.4 Warehouse / Multi-Location Mapping
- **Proposed:** `warehouses` table; replace single `stock_quantity` with `inventory_levels` (product/variant √ó warehouse ‚Üí qty, reserved, available). Keep aggregate `stock_quantity` as a view/derived column for backward compat.
- **Importance:** Multi-warehouse = faster fulfillment, accurate ETAs, regional stock.

### 3.5 Category Hierarchy Management
- **Proposed:** Use existing `product_categories` (already referenced by `category_attributes`); add `parent_id` self-FK for nested trees, `slug`, `position`, `is_active`. Backfill `products.category` TEXT into the FK.
- **Importance:** Navigation, faceted search, and merchandising depend on a real taxonomy.

**See Phase 1 section for concrete migrations, services, routes, and frontend pages.**

## 4. Business Model Alignment ‚ÄĒ Single-Vendor (REQUIRED)

**Objective:** Ensure the entire architecture, schema, and workflows are designed strictly for a **Single-Vendor** e-commerce model. Remove any multi-vendor / marketplace capabilities.

### 4.1 Current State
- The platform is **already single-tenant**: `stores` is constrained to a single record (`060_single_store_hardcode.sql`), and all catalog/order/inventory data is scoped to that one store. This aligns with the single-vendor requirement ‚ÄĒ **no storefront multi-tenancy to remove**.
- The term `onboarding` in the codebase refers to the **customer questionnaire** (`onboarding_questions`/`onboarding_user_answers`, `002_onboarding.sql`), **not seller/vendor onboarding**. It is retained (it is buyer onboarding, not marketplace seller onboarding).
- Minor residue to clean up: `stores.business_type` CHECK still allows `'marketplace'` (`053_create_stores_table.sql`); the `roles` seed comment references a `'vendor'` example (`004_create_roles_tables.sql`).

### 4.2 Removals / Constraints
- **No vendor dashboards, no seller portals, no vendor management UI.**
- **No split payments / payouts / settlement / commission** logic. All order revenue belongs to the single store; payments are single-destination (existing `payment.pay_on_delivery_enabled` + gateways suffice).
- **No seller onboarding / seller KYC / seller approval workflows.**
- Constrain `stores.business_type` CHECK to `('retail','wholesale','dropshipping','other')` ‚ÄĒ **drop `'marketplace'`**.
- Remove the `'vendor'` role reference from the `roles` table comment; ensure no `vendor*` permission keys are seeded.
- Pricing, inventory, orders, and analytics all implicitly belong to the single store (no `vendor_id` partitioning needed).

### 4.3 Functional Importance
A single-vendor model removes an entire settlement/compliance/tenant-isolation layer, drastically simplifying the data model, accelerating time-to-market, and eliminating payout/commission bugs that plague marketplaces. The admin portal becomes a pure first-party operations console.

## 5. Order & Lifecycle Management

**Objective:** End-to-end, state-machine-driven order flow with financial documents and dispute handling.

### 5.1 Order Status State Machine
- **States:** `PENDING ‚Üí PROCESSING ‚Üí SHIPPED ‚Üí DELIVERED`, with `RETURNED`, `CANCELLED`, `REFUNDED`. Model as explicit transition table `order_status_transitions` enforced server-side (no arbitrary jumps).
- **Importance:** Prevents invalid transitions (e.g., shipping a cancelled order) that cause financial/reputation damage.

### 5.2 Automated Invoicing
- **Proposed:** `invoices` table + `invoice.service.js` generating PDF/invoice numbers with **gross NGN totals only** (no tax lines ‚ÄĒ see ¬ß1.3), sent via notification bus. Triggered on `DELIVERED`/`PAID`.
- **Importance:** Legal compliance + easier reconciliation for finance/audit roles.

### 5.3 Refund / Dispute Management
- **Proposed:** `refunds` (amount, reason, method, status), `disputes` (buyer‚ÜĒstore/merchant, evidence, SLA timers), approval gates (`finance:approve`). Reuse `022_orders_refunds_returns` base. (Single-vendor: disputes are between customer and the store, not between buyers and third-party sellers.)
- **Importance:** Disputes are the top churn + chargeback driver; SLA tracking protects seller rating.

**Endpoints:** `POST /admin/orders/:id/transition`, `GET /admin/orders/:id/invoice`, `POST /admin/refunds`, `POST /admin/disputes/:id/resolve`.

### P1.1 Database Migrations
New file `Backend/sql/076_catalog_inventory_governance.sql`:

```sql
-- 1. Category hierarchy (use existing product_categories, add tree + slug)
ALTER TABLE product_categories
  ADD COLUMN IF NOT EXISTS parent_id UUID REFERENCES product_categories(id) ON DELETE SET NULL,
  ADD COLUMN IF NOT EXISTS slug TEXT UNIQUE,
  ADD COLUMN IF NOT EXISTS position INT DEFAULT 0,
  ADD COLUMN IF NOT EXISTS is_active BOOLEAN DEFAULT TRUE;
CREATE INDEX IF NOT EXISTS idx_pc_parent ON product_categories(parent_id);

-- 2. Variant option matrix
CREATE TABLE IF NOT EXISTS product_options (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  name TEXT NOT NULL,                 -- "Size", "Color"
  display_order INT DEFAULT 0,
  UNIQUE (product_id, name)
);
CREATE TABLE IF NOT EXISTS product_option_values (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  option_id UUID NOT NULL REFERENCES product_options(id) ON DELETE CASCADE,
  value TEXT NOT NULL,
  position INT DEFAULT 0,
  UNIQUE (option_id, value)
);
-- product_variants.option_values becomes derived via join table:
CREATE TABLE IF NOT EXISTS product_variant_options (
  variant_id UUID NOT NULL REFERENCES product_variants(id) ON DELETE CASCADE,
  option_value_id UUID NOT NULL REFERENCES product_option_values(id) ON DELETE CASCADE,
  PRIMARY KEY (variant_id, option_value_id)
);

-- 3. Warehouses + multi-location stock
CREATE TABLE IF NOT EXISTS warehouses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  location TEXT,
  is_active BOOLEAN DEFAULT TRUE
);
CREATE TABLE IF NOT EXISTS inventory_levels (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID REFERENCES products(id) ON DELETE CASCADE,
  variant_id UUID REFERENCES product_variants(id) ON DELETE CASCADE,
  warehouse_id UUID NOT NULL REFERENCES warehouses(id) ON DELETE CASCADE,
  quantity INT NOT NULL DEFAULT 0,
  reserved INT NOT NULL DEFAULT 0,
  low_stock_threshold INT DEFAULT 10,
  UNIQUE (product_id, variant_id, warehouse_id)
);
-- keep aggregate for backward compat (maintained by trigger from inventory_levels)
CREATE OR REPLACE FUNCTION sync_product_stock() RETURNS trigger AS $$
BEGIN
  UPDATE products p SET stock_quantity = (
    SELECT COALESCE(SUM(quantity - reserved),0) FROM inventory_levels il
    WHERE il.product_id = p.id)
  WHERE p.id = COALESCE(NEW.product_id, OLD.product_id);
  RETURN NEW;
END; $$ LANGUAGE plpgsql;
DROP TRIGGER IF EXISTS trg_sync_product_stock ON inventory_levels;
CREATE TRIGGER trg_sync_product_stock AFTER INSERT OR UPDATE OR DELETE ON inventory_levels
  FOR EACH ROW EXECUTE FUNCTION sync_product_stock();

-- 4. Stock alert rules
CREATE TABLE IF NOT EXISTS stock_alert_rules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scope TEXT CHECK (scope IN ('product','variant','warehouse','global')) DEFAULT 'product',
  threshold INT NOT NULL,
  channel TEXT[] DEFAULT '{in_app,email}',
  recipient_role TEXT,
  is_active BOOLEAN DEFAULT TRUE
);

-- 5. Bulk import jobs
CREATE TABLE IF NOT EXISTS import_jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_type TEXT NOT NULL CHECK (entity_type IN ('product','variant','inventory','category')),
  status TEXT NOT NULL CHECK (status IN ('queued','processing','completed','failed','partial')),
  total_rows INT DEFAULT 0,
  processed_rows INT DEFAULT 0,
  error_rows INT DEFAULT 0,
  error_file_url TEXT,
  created_by UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```
