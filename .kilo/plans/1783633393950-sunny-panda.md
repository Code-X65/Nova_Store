# Plan: Convert Multi-Store to Single-Store System

## Goal

Convert the Nova Store backend + admin frontend from a multi-store capable architecture to a true single-store system, while **preserving all `store_id` data tracking** on every entity.

## Current State

- Database has a `stores` table (one row: `slug='nova-store'`)
- Every entity table (`products`, `orders`, `users`, `carts`, etc.) has a `store_id UUID` column with NOT NULL + indexes
- Backend resolves store context from `req.user.store_id` or `req.admin.store_id` via `storeContext` middleware
- `scopeToStore` enforces per-user store ownership
- Frontend `StoreSwitcher` is display-only; there is no store-switching UI or API
- Only one store is ever created; no multi-store admin features exist

## Approach

Hardcode the single store ID throughout the backend, simplify middleware, and clean up the frontend. **Keep `store_id` columns, indexes, and the `stores` table** so all historical and future data remains store-tagged.

### 2.2 Services — optional cleanup

- **A (minimal):** Keep parameter, pass `SINGLE_STORE_ID` from controllers

- `product.service.js` — remove `storeId` param, hardcode in `createProduct`, `updateProduct`, `addRelatedProduct`, etc.
- `order.service.js` — remove `storeId` param, hardcode in `getUserOrders`, `getOrderDetails`, etc.

- `category.service.js` — remove `storeId` param

## Key Decisions Needed

1. **`scopeToStore` middleware:** Remove entirely or keep as silent no-op?
   - **Recommendation:** Keep as no-op. Defense-in-depth; zero runtime cost.

2. **Service `storeId` parameter:** Remove from function signatures or keep passing `SINGLE_STORE_ID`?
   - **Recommendation:** Remove from ~7 key services (product, order, inventory, cart, checkout, category, invitation) to reduce noise. Keep in store model/service since they legitimately manage store data.

3. **Frontend `AdminSession.storeId`/`storeName`:** Keep in types or remove?
   - **Recommendation:** Keep. The backend still returns them; removing breaks the API contract for no benefit.

4. **Environment variable vs. hardcoded UUID for `SINGLE_STORE_ID`:**
   - **Recommendation:** Use `process.env.SINGLE_STORE_ID` with a fallback query to find the `nova-store` slug at startup. This works across dev/staging/prod.

5. **Database migration needed?**
   - **Recommendation:** Yes, a lightweight migration to assert all `store_id` columns are populated and add a comment noting single-store mode. Do NOT drop columns.
