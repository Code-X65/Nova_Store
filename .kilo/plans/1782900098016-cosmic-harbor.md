# Nova Store Backend Architecture Report

## Executive Summary

The Nova Store backend is engineered as a **multi-vendor marketplace system** (NOT single-store as suggested), featuring a comprehensive RBAC (Role-Based Access Control) system with two distinct user phases: **Customer-facing operations** and **Administrative operations**.

## Confirmation of User's Assertions

**Partially Correct:** The system does have a built-in "superadmin" (SUPER_ADMIN role) that can invite other admins. However, this is more nuanced than a simple single-store setup.

**Incorrect:** The backend is NOT "highly setup for a single store" - it supports multi-vendor operations with products, categories, brands, and full e-commerce functionality.

## Complete Backend Architecture

### 1. User Authentication & Identity System

#### Customer Authentication (`/api/v1/auth`)
- **Registration**: `POST /register` - Customers sign up with email, password, phone verification (OTP), home address, and referral source
- **Login**: `POST /login` - JWT-based auth with session cookies, refresh token rotation
- **OAuth Providers**: Google, Facebook, Apple integrations (OAuth 2.0)
- **Password Management**: Forgot/reset password flow with email tokens
- **Verification**: Email verification tokens + phone OTP verification
- **Rate Limiting**: Auth-specific rate limiting for login, registration, password reset

#### Admin Authentication (`/api/v1/admin`)
- **Login**: `POST /admin/login` - Session-based authentication with 8-hour token expiry (vs 30-day for customers)
- **Stricter Lockout**: 3 failed attempts triggers 60-minute lockout (vs 5 attempts / 15 min for customers)
- **Session Management**: Max 3 concurrent admin sessions enforced
- **Verification**: `GET /admin/verify` - Session validation endpoint

#### User Model (`users` table)
- Core fields: email, password_hash, first_name, last_name, phone_number
- Role field: `USER`, `ADMIN`, `SUPER_ADMIN`, `MODERATOR`, `CUSTOMER`
- Status fields: `is_email_verified`, `is_active`, `is_locked`, `failed_login_attempts`
- Optional fields: referral_code, membership_level (BRONZE/SILVER/GOLD/PLATINUM), loyalty_points
- Feature flags: analytics, collaboration, api_access, advanced_reporting
- Onboarding state tracking: `onboarding_status` (not_started/in_progress/completed/skipped)

### 2. Role-Based Access Control (RBAC)

#### Roles Table Structure
| Role | Description | Permissions |
|------|-------------|-------------|
| `SUPER_ADMIN` | Full system access, can invite/manage all admins | All (*) + wildcard permission |
| `ADMIN` | Full administrative access (deprecated, now via user_roles) | All permissions |
| `MODERATOR` | Content/user moderation | user:read, product:write, order:read/write, admin:access |
| `CUSTOMER` (default) | Regular user access | user:read (own data only) |

#### User-Roles Relationship
- Users can have multiple roles via `user_roles` junction table
- Granular permissions can be assigned per-user via `extra_permissions` JSONB field
- Permission middleware checks both role-based and user-specific permissions

#### Key Permissions Include:
- `user:read`, `user:write`, `user:delete` - User management
- `product:create`, `product:write`, `product:delete` - Product management
- `order:read`, `order:write` - Order management
- `inventory:read`, `inventory:write`, `inventory:alert` - Stock management
- `role:manage`, `permission:assign` - RBAC management
- `*` - Wildcard (FULL SYSTEM ACCESS)

### 3. Admin Invitation System

#### Invitation Flow
1. **SUPER_ADMIN creates invitation** via `POST /api/v1/admin/invitations`
   - Validates inviter is SUPER_ADMIN
   - Checks email not already registered
   - Checks no pending invitation exists
   - Creates 64-char hex token, sets 7-day expiry (configurable)
   - Sends invitation email with accept link

2. **Admin accepts invitation** via public endpoints:
   - `GET /api/v1/accept-invite/{token}` - Validates token, returns email/role/expiry
   - `POST /api/v1/accept-invite/{token}` - Creates user account, assigns role

#### Invitation Model
- Status tracking: `pending`, `accepted`, `expired`, `revoked`
- Linked to inviter, role, and optional extra permissions
- Cleanup job expires stale invitations automatically

### 4. Customer Operations Phase

#### Product Browsing & Discovery
- `GET /api/v1/products` - List products with filters (category, brand, price, search, attributes)
- `GET /api/v1/products/{slug}` - Product detail with variants
- `GET /api/v1/products/featured` - Featured products
- Full-text search via `search_products` RPC
- Attribute-based filtering via `get_products_by_attributes` RPC

#### Shopping Cart
- **Guest & Authenticated Support**: Carts tied to userId OR sessionId
- **Stock Validation**: Real-time stock checking, backorder support
- Endpoints:
  - `GET /api/v1/cart` - Retrieve cart
  - `POST /api/v1/cart` - Add item
  - `PUT /api/v1/cart/items/:id` - Update quantity
  - `DELETE /api/v1/cart/items/:id` - Remove item
  - `POST /api/v1/cart/merge` - Merge guest cart to user account after login

#### Wishlist
- `GET /api/v1/wishlist` - Get user's wishlist
- `POST /api/v1/wishlist` - Add/remove items

#### Checkout & Payments
- `POST /api/v1/checkout` - Create checkout session
- `POST /api/v1/payments` - Process payments (Paystack, Flutterwave, Stripe)
- Idempotent checkout prevents duplicate orders

#### Orders
- **Customer**: View orders, cancel, request returns, reorder
- **Guest Orders**: Claim via verified email address
- **Status Flow**: pending → confirmed → processing → shipped → delivered (with cancellation/return paths)
- Return workflow: customer requests → admin processes → refund handled

### 5. Administrative Operations Phase

#### SuperAdmin-Only Capabilities
- **Admin Management**: List, view, update roles/permissions, revoke access
- **Invitation Management**: Send, list, revoke, resend invitations
- **System Settings**: Configure store-wide settings
- **Analytics**: Telemetry and reporting views

#### Admin-Accessible Features
- **Product Management**: CRUD operations (categories, brands, attributes, variants)
- **Order Management**: Full order lifecycle (dispatch, delivery, returns)
- **Inventory**: Stock adjustments, low-stock alerts, reservation system
- **Coupons**: Create/manage discount codes
- **Reviews**: Moderate product reviews
- **Notifications**: Email/SMS notification templates and logs
- **Shipping**: Shipping zones, rates, delivery dispatch

### 6. Database Schema Overview

#### Core Tables
- `users` - Customer/admin users (RLS disabled, backend-managed)
- `roles` - Role definitions
- `permissions` - Granular permission definitions
- `user_roles` - Many-to-many user-role assignments
- `role_permissions` - Many-to-many role-permission assignments
- `invitations` - Admin invitation tokens
- `products` - Product catalog with variants support
- `product_variants` - SKU variants (size, color, etc.)
- `categories` / `brands` - Product taxonomy
- `carts` / `cart_items` - Shopping cart
- `wishlists` / `wishlist_items` - Wishlist/favorites
- `orders` / `order_items` - Order management
- `payments` - Payment transactions
- `inventory_transactions` - Stock change history
- `sessions` - User sessions with refresh tokens
- `admin_sessions` - Admin-specific sessions
- `password_resets` / `verification_tokens` - Email verification/reset

### 7. Security Features

- **JWT Authentication**: Access + refresh token rotation
- **CSRF Protection**: Session-based CSRF tokens
- **Rate Limiting**: Endpoint-specific rate limits (auth, admin, API)
- **Input Sanitization**: XSS sanitization middleware
- **Password Policy**: 12+ chars, upper/lower/number/special required
- **Account Lockout**: Progressive lockout (5 attempts / 15 min for customers, 3 attempts / 60 min for admins)
- **Session Security**: HttpOnly, Secure, SameSite cookies
- **Redis Session Store**: High-performance session caching
- **Request ID & Audit Logging**: All actions logged with traceability

### 8. Key Business Logic Distinctions

| Aspect | Customer | Admin |
|--------|----------|-------|
| Token Expiry | 15min access / 30d refresh | 15min access / 8h refresh |
| Concurrent Sessions | Unlimited | Max 3 |
| Registration | Self-service | Invite-only |
| Registration Status | Pending email verification | Email verified on invite acceptance |
| Lockout Policy | 5 failed / 15 min | 3 failed / 60 min |

### 9. Missing Elements (Potential Limitations)

- **No Store/Vendor Isolation**: Products are global; no per-store scoping
- **No Multi-Tenancy**: Single database schema without tenant separation
- **Single Currency**: Appears to support only one currency (though column exists)
- **Single Country Focus**: Default country set to Nigeria throughout

### Conclusion

The backend is a **comprehensive e-commerce platform** with:
1. A **robust RBAC system** enabling multi-admin management via invitations
2. **Two-phase user experience**: Customers self-register; Admins are invited by SUPER_ADMIN
3. **Full e-commerce features**: Products, cart, wishlist, checkout, orders, payments, returns
4. **Administrative controls**: Product management, order fulfillment, inventory, analytics

This is a **multi-vendor capable marketplace backend** rather than a single-store system, with sophisticated admin onboarding through invitation tokens.

# Plan: Convert to Single-Store Architecture with Future Multi-Store Support

## Objective
Transform the current multi-vendor marketplace into a single-store system where:
- **SUPER_ADMIN = Store Owner** (1 per store)
- **ADMIN = Store Staff** (multiple per store)
- All entities scoped by `store_id` for future multi-store expansion

## Phase 1: Database Schema Updates (New Migration)

### 1.1 Create Stores Table (`sql/053_create_stores_table.sql`)
```sql
CREATE TABLE stores (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  name TEXT NOT NULL,
  slug TEXT UNIQUE NOT NULL, -- e.g., 'nova-store-main'
  email TEXT, -- store contact email
  phone TEXT,
  address JSONB, -- full address object
  timezone TEXT DEFAULT 'UTC',
  currency TEXT DEFAULT 'NGN',
  is_active BOOLEAN DEFAULT TRUE,
  created_by UUID NOT NULL REFERENCES users(id), -- The SUPER_ADMIN who created it
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Store settings (can be extended later)
CREATE TABLE store_settings (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  store_id UUID NOT NULL REFERENCES stores(id) ON DELETE CASCADE,
  key TEXT NOT NULL,
  value JSONB NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  UNIQUE(store_id, key)
);

ALTER TABLE stores DISABLE ROW LEVEL SECURITY;
ALTER TABLE store_settings DISABLE ROW LEVEL SECURITY;
```

### 1.2 Add `store_id` to Existing Tables (`sql/054_add_store_id_to_tables.sql`)
Tables requiring `store_id` column addition:
- `users` - Admin/staff users belong to a store
- `products`, `product_variants` - Store inventory
- `product_categories`, `product_brands` - Store taxonomy
- `orders`, `order_items` - Store orders
- `carts`, `cart_items` - Store shopping carts
- `wishlists`, `wishlist_items` - Store wishlists
- `coupons` - Store coupons
- `categories` (if separate from product_categories)
- `brands` (if separate from product_brands)
- `inventory_transactions` - Store stock movements
- `addresses` - Customer addresses (store context for analytics)

### 1.3 Create Default Store and Migrate Existing Data
- Generate a UUID for the default store
- Seed `store_id` into all existing records
- Link the current `admin@novastore.com` SUPER_ADMIN as the store owner

## Phase 2: Model Updates

### 2.1 Update User Model (`src/models/user.model.js`)
- Add `store_id` to user creation/updates
- Filter queries by `store_id` when user is admin/staff
- SUPER_ADMIN sees all stores; ADMIN sees only their store

### 2.2 Update Product Model (`src/models/product.model.js`)
- Add `store_id` to all product queries
- Filter products by store in findAll, search, featured

### 2.3 Update All Related Models
- `category.model.js`, `brand.model.js` - store-scoped taxonomy
- `order.model.js`, `cart.model.js`, `wishlist.model.js` - store context
- `inventory.service.js` - stock operations scoped to store

## Phase 3: Middleware & Context

### 3.1 Create Store Context Middleware (`src/middlewares/store-context.middleware.js`)
```javascript
// Attaches req.store based on:
// 1. User's store_id (for logged-in users)
// 2. Default store for unscoped operations
const storeContext = async (req, res, next) => {
  if (req.user?.id) {
    req.store = await storeModel.findUserStore(req.user.id);
  } else {
    req.store = await storeModel.getDefaultStore();
  }
  next();
};
```

### 3.2 Update RBAC for Store Context
- Modify `require-admin.middleware.js` to include store in `req.admin`
- SUPER_ADMIN: Can view multiple stores (if created)
- ADMIN: Restricted to their store's data
- Add `ownsStore(user, storeId)` helper for ownership checks

## Phase 4: Service Layer Changes

### 4.1 Invitation Service Updates
- Invited admins automatically assigned to inviter's store
- Store context in invitation creation
- Prevent cross-store admin invitations

### 4.2 Product/Category/Brand Services
- All CRUD operations automatically scope to `req.store.id`
- List endpoints filter by store_id

### 4.3 Order/Cart/Wishlist Services
- Orders tied to store context
- Future: Customers see products from "their" store

## Phase 5: API Scope Restrictions

### 5.1 Admin Routes (`/api/v1/admin`)
- Add middleware chain: requireAdmin → storeContext → scopeToStore
- All admin endpoints filter to their store
- Categories/brands/products scoped to store

### 5.2 Product Routes (`/api/v1/products`)
- Products visible based on store context
- Future: Customer API could filter by store subdomain/header

## Key Design Decisions

### Option A: Hardcoded Default Store (Recommended for MVP)
- Single store UUID hardcoded in environment
- All operations implicitly scoped to this store
- Simple, no store selection UI needed
- Easy migration path to multi-store later

### Option B: Store Selection Middleware
- Dynamic store resolution via subdomain/header
- More complex but forward-compatible
- Can be implemented later when needed

## Questions for Clarification

1. **Should SUPER_ADMIN be renamed to `STORE_OWNER`?** Or keep as-is with store ownership semantics?
2. **Store Staff Permissions**: Should ADMIN staff have all permissions, or should we create granular staff roles (inventory_staff, order_staff, etc.)?
3. **Customer Scope**: When multi-store launches, should customers be tied to a specific store, or can they browse all stores?
4. **Migration Strategy**: Should we run migrations automatically, or create a manual setup script for the store owner to configure their store?
5. **Store Branding**: Should the store have configurable branding (logo, colors) visible on the frontend?
