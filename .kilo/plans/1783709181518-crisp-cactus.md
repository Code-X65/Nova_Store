# Nova Store — Dual-Purpose Notification & Audit Logging System
### Technical Specification + Implementation Roadmap

> **Scope:** A single spec covering the Audit Trail (immutable, compliance-grade) and the Notification system (real-time, transient), for both the **Admin Portal frontend** and the **Backend architecture**.
>
> **Authoring basis:** Written against the actual current Nova Store codebase (read-only audit, 2026-07-10). Where the requested design overlaps existing infrastructure, the plan reuses it rather than duplicating. Net-new work is scoped to the gaps.

---

### Critical gaps this plan fills
1. **Audit schema** lacks `severity`, actor `full_name`/`role`/`session_id`, and a typed `action_type` enum. `action` is a free-text string.
2. **Frontend/Backend mismatch bug:** `AuditLogs.tsx` reads `entity_type`/`entity_id` (lines 47-53) but the model writes `resource_type`/`resource_id` (`audit-log.model.js:8-9`) → UI always shows blank "Entity".
3. **No field-level human delta** (e.g., "Price changed from $10 to $12") — only raw JSONB `old_values`/`new_values`.
4. **No role/team-based routing engine** — notifications are user-id targeted only. Admin team alerts (Sales/Warehouse) are absent.
5. **No automatic domain alerts** for: catalog critical deletes / bulk attribute changes, new order to Sales, payment failed, out-of-stock during picking, order shipped, low/out-of-stock, inventory discrepancy, new review (rating-based).
6. **SSE gateway not wired to notifications** → frontend must poll; no instant popups.
7. **Audit dashboard lacks** free-text search, severity/action/actor filters, and **CSV/PDF export**.
8. **No audit middleware/interceptor** to auto-enrich actor context (calls are manual).

### Confirmed architectural decisions
- **Realtime transport:** Extend the existing **SSE** gateway (do **not** add Socket.io).
- **Event bus:** Reuse **Redis Pub/Sub** (add a typed in-process EventEmitter + Redis channels; no RabbitMQ/Kafka).

## 1. Core Data Architecture

### 1.1 Unified Audit Event schema (extension of `audit_logs`)
New migration `Backend/sql/063_audit_enrichment.sql` alters `audit_logs` to capture everything in the spec while staying append-only and back-compatible:

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | unchanged |
| `event_id` | TEXT UNIQUE | NEW — external correlation id (ULID) for idempotency/dedup |
| `severity` | TEXT | NEW — `info` \| `warning` \| `critical` (default `info`) |
| `action_type` | TEXT | NEW — enum `CREATE` \| `UPDATE` \| `DELETE` \| `LOGIN` \| `STATUS_CHANGE` \| `OTHER` |
| `action` | TEXT | kept (e.g. `product.updated`) — now normalized |
| `resource_type` | TEXT | kept (= "Entity Type"), aliased to `entity_type` in API response |
| `resource_id` | UUID | kept (= "Entity ID"), aliased to `entity_id` |
| `actor_user_id` | UUID | rename of `user_id` (or keep, add below) |
| `actor_full_name` | TEXT | NEW — denormalized snapshot |
| `actor_role` | TEXT | NEW — denormalized role at event time |
| `actor_session_id` | TEXT | NEW — from `req.session?.id` / `req.admin` session |
| `ip_address` | INET | kept |
| `user_agent` | TEXT | kept |
| `request_id` | TEXT | kept (from `request-id.middleware`) |
| `old_values` | JSONB | kept (Before state) |
| `new_values` | JSONB | kept (After state) |
| `delta` | JSONB | NEW — field-level diff `[{field, before, after, label}]` |
| `summary` | TEXT | NEW — human-readable sentence (notifications reuse this) |
| `created_at` | TIMESTAMPTZ | kept |

> **Back-compat:** API layer maps `resource_type` to `entity_type`, `resource_id` to `entity_id`. `AuditLogs.tsx` will then render correctly. Old rows keep `severity='info'`, `action_type='OTHER'`.

### 1.2 Notification schema (reuse existing; add routing)
Existing `notifications` table already matches the spec (type, title, message, data JSONB = payload w/ deep-link, channel[], status, read_at). Additions in `064_notification_routing.sql`:
- `notifications.recipient_role` TEXT — for team-routed admin alerts (nullable; NULL = direct user).
- `notifications.severity` TEXT — mirrors audit severity for UI styling.
- `notification_routing_rules` table: `{ id, name, event_key, recipient_role(s)[] (team), channel[], severity, is_active }` — the **Role-Based Routing Engine** config.
- Teams map to existing RBAC roles: Sales Team → `ORDER_STAFF`, Warehouse Team → `INVENTORY_STAFF`, Owners → `STORE_OWNER` (no new team table).

### 1.3 Delta / summary helper
`Backend/src/utils/audit-diff.js` (new): `computeDelta(oldObj, newObj, fieldLabels)` → `{ delta, summary }`.
- Produces `delta[]` of changed shallow + nested fields.
- Renders `summary` e.g. `"Price changed from $10 to $12; Status: draft → published"`.
- Used by all UPDATE audit calls (replaces ad-hoc `oldValues`/`newValues` passing).

## 2. Backend Engineering (The Engine) — Phase 5

### 2.1 Actor-context middleware (the "Interceptor")
**New `Backend/src/middlewares/audit-context.middleware.js`:**
- Runs after `requireAdmin`/`protect`. Attaches `req.actor = { id, fullName, role, sessionId }`.
- `fullName` from `req.admin`/`req.user` profile; `role` from primary role; `sessionId` from session cookie / `req.admin.sessionId`.
- `AuditService.log` reads `req.actor` to populate `actor_*` columns automatically.

### 2.2 Domain Event Bus
**New `Backend/src/realtime/event-bus.js`:**
- In-process `EventEmitter` (`emit(domainEvent)`, `on(eventType, handler)`).
- On emit, also `redisClient.publish('nova:events', JSON.stringify(envelope))`; a subscriber (`nova:events`) replays to other instances for horizontal scale.
- Typed events: `catalog.product.created`, `catalog.product.deleted`, `catalog.attribute.bulk_changed`, `order.placed`, `order.payment_failed`, `order.picked_out_of_stock`, `order.shipped`, `inventory.low_stock`, `inventory.out_of_stock`, `inventory.discrepancy`, `review.created`, `staff.permission_changed`, `staff.role_escalated`, `staff.user_created`.

### 2.3 Audit middleware wrapper (declarative auto-capture)
**New `Backend/src/middlewares/audit-action.middleware.js`:**
- `auditAction({ actionType, resourceType, severity, opts })` higher-order middleware for generic CRUD auto-logging. Computes delta from `req.beforeState` (set by controllers before mutate) vs result. Reduces manual `AuditService.log` boilerplate.
- Existing manual calls stay (they carry rich context); new generic endpoints use the wrapper.

### 2.4 Notification Routing Engine
**New `Backend/src/services/notification-router.service.js`:**
- `route(event)` → looks up `notification_routing_rules` by `event_key`, resolves recipient **user ids per role/team** (`resolveTeamRecipients(roles[])`), and for each recipient calls `NotificationService.sendToUser` with template + `recipient_role` + `severity`.
- Publishes each created in-app notification to SSE (`publishToSse(userId, event)` → `nova:rt:user:<id>`), so the admin bell updates instantly.
- Falls back to existing per-user `sendToUser` for customer notifications (unchanged path).

### 2.5 Wire domain events to handlers
Add thin handlers (in respective services/controllers) that `eventBus.emit(...)`:
- **Catalog (Phase 1):** `product.controller.js` emit `catalog.product.deleted` (severity `critical` if not soft-archive) + `catalog.attribute.bulk_changed` when `POST /categories/bulk` or attribute bulk ops exceed threshold → route to `MANAGER`/`STORE_OWNER` team with "possible accidental catalog wipe" alert.
- **Orders (Phase 2):** `order.service.js`/`checkout.service.js` emit `order.placed` to Sales Team; `payment.service.js` emit `order.payment_failed`; picking step emit `order.picked_out_of_stock`; `order.controller.js` `order.shipped` to Warehouse/Sales. Each writes an audit `STATUS_CHANGE` with summary `"Status changed from Pending to Shipped by Admin [Name]"`.
- **Inventory (Phase 3):** threshold breach detector (new cron or on `adjustStock`/`reduceStock`) emits `inventory.low_stock` / `inventory.out_of_stock`; `adjustStock` with `reasonCode` + discrepancy emits `inventory.discrepancy` (already audited at `inventory.controller.js:70`).
- **Reviews/Users (Phase 4):** `review.service.js:89` emit `review.created` with `severity = rating<=2 ? critical : rating>=4 ? info(marketing) : info`; `admin-management.controller.js` + `invitation.service.js` emit `staff.permission_changed`/`staff.role_escalated`/`staff.user_created` (security-critical, severity `critical`, route to `STORE_OWNER`).

### 2.6 Database design & retention
- Keep `audit_logs` (append-only, high-volume) and `notifications` (transient) **separate** — already the case.
- Retention policy: archive `audit_logs` older than 18 months to `audit_logs_archive` via existing cron (`Backend/src/jobs/`); purge `read_at` notifications older than 90 days. New `cleanup.job.js` additions.
- Indexes: add `idx_audit_logs_severity`, `idx_audit_logs_action_type`, `idx_audit_logs_actor` (migration 063).

## 3. Functional Requirements by Module (Phases 1-4)

### Phase 1 — Catalog Management (Audit Focus)
- **Already audited:** product/category/brand/attribute CRUD (`product.controller.js`, `product-category.controller.js`, `product-brand.controller.js`, `attribute.controller.js:54-145`).
- **Gap to build:** enrich those calls with `action_type`, `delta`, `summary` (via `audit-diff`). Emit `catalog.product.deleted` (critical) + `catalog.attribute.bulk_changed` notifications to Manager/Owner team when: hard delete, or bulk attribute change count > N (configurable, default 20) → "Possible accidental catalog wipe".

### Phase 2 — Order Lifecycle (Notification & Workflow Focus)
- **Already:** customer `order_created` (checkout.service.js:179); order status audit (order.controller.js:113-276).
- **Gap to build:** Role-Based Routing Engine routes:
  - `order.placed` → **Sales Team** (`ORDER_STAFF`) — "New Order #X".
  - `order.payment_failed` → Sales Team (warning).
  - `order.picked_out_of_stock` → Warehouse Team (`INVENTORY_STAFF`) (critical).
  - `order.shipped` → Warehouse + Sales (info).
  - Audit `STATUS_CHANGE` summary: `"Status changed from Pending to Shipped by Admin [Name]"` (use `req.actor.fullName`).

### Phase 3 — Inventory & Stock Control
- **Already:** manual adjustments audited with `reasonCode` (inventory.controller.js:70; route accepts `reasonCode` at :49).
- **Gap to build:** threshold listeners emit `inventory.low_stock` / `inventory.out_of_stock` (use existing `get_low_stock_products` RPC + `/inventory/alerts` config); `inventory.discrepancy` when `adjustStock` reason is a manual correction. Route to Warehouse/Inventory team.

### Phase 4 — Customer Engagement & User Management
- **Reviews:** `review.service.js:89` creates review but emits no audit/notification. **Build:** emit `review.created`; severity `critical` if rating ≤ 2 (flag for support), `info`/marketing if rating ≥ 4. Route low-rating to Sales/Support team.
- **Teams & Roles:** permission changes / new users / role escalations partially audited (admin-management.controller.js:155,220; invitation.service.js). **Build:** dedicated `staff.permission_changed`/`staff.role_escalated`/`staff.user_created` events, severity `critical`, routed to `STORE_OWNER` (security-critical). Ensure `old_values` captures prior permission/role set.

## 7. Key Decisions / Tradeoffs (locked)
- Reuse **SSE** (not Socket.io) and **Redis Pub/Sub** (not RabbitMQ/Kafka). No new infra.
- Teams = existing RBAC roles (no new `notification_teams` table) unless a dedicated team abstraction is later required.
- Audit remains append-only; `old_values`/`new_values` retained alongside new `delta`/`summary` for compliance replay.
