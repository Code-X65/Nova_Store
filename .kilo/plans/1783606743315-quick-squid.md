# Catalog Gap Analysis — CRUD · Notifications · Audit Log (Frontend + Backend)

> Read-only gap analysis. Scope: the **Catalog** domain (Products, Categories, Brands, Attributes) and the cross-cutting **Notification** and **Audit Log** capabilities. Source of truth: backend at `Backend/src`, frontend admin at `Frontend/src/admin`, seeded schema/permissions at `Backend/sql`.

---

## 0. Executive Summary

| Area | Status | Headline gap |
|------|--------|--------------|
| **Catalog CRUD** | ✅ Largely complete | Permission-key alignment now fixed by seeds `058/059`; minor: no standalone Attributes route, no catalog API service layer, create buttons not gated by `*:create`. |
| **Notifications** | ⚠️ Partial | **No catalog-driven notifications at all** (incl. low-stock alerting, which is seeded but unwired). Broadcast/email fan-out stubbed. Real-time is polling-only. |
| **Audit Log** | ⚠️ Partial | Catalog CRUD **is** audited backend-side, **but** the admin Audit UI references non-existent fields (`entity_type`/`entity_id`), so catalog entries render with a blank Entity column, and auth tabs query the wrong table shape. No diff viewer, no stats usage, audit failures silently swallowed. |

Overall: **backend CRUD + audit capture is solid; the weak spots are (1) catalog→notification wiring, (2) audit-log UI rendering/UX, and (3) frontend catalog structure polish.**

## 1. Catalog CRUD

### 1.1 Backend — present ✅
- **Products** (`product.routes.js`, `product.controller.js`, `product.service.js`, `product.model.js`): GET list/by-id/slug/stock/search/price-range/featured/recommendations/related (all **public**); `POST /products`, `POST /products/bulk`, `PATCH /products/:id`, `DELETE /products/:id`; images (`POST/DELETE /products/:id/images`), variants (`POST/PUT/DELETE /products/:id/variants/:variantId`), related (`POST/DELETE /products/:id/related/:relatedId`). Guards: `product:create` (POST/bulk), `product:write` (PATCH/images/variants/related), `product:delete` (DELETE/variants). All mutations pass `scopeToStore`.
- **Categories** (`product-category.routes.js` + `category.service.js`): GET tree/by-id/slug/subcats (public); `POST /categories`, `POST /categories/bulk`, `PUT /categories/reorder`, `PATCH /categories/:id`, `DELETE /categories/:id`. Guards `category:create` / `category:write`.
- **Brands** (`product-brand.routes.js`, no service — controller hits model directly): GET list/by-id/slug (public); `POST /brands`, `PATCH /brands/:id`, `DELETE /brands/:id`. Guards `brand:create`/`brand:write`/`brand:delete`.
- **Attributes** (`attribute.routes.js` + `attribute.service.js`): `GET /categories/:id/attributes` (public); `POST /categories/:id/attributes`, `PUT /attributes/:attributeId`, `DELETE /attributes/:attributeId`. Guard `category:write`.
- **Relations**: migrations `031` (products.category_id FK) and `039` (category/brand product-count triggers, attribute-value validation) keep counts in sync. Service validates category_id/brand_id existence before write.

### 1.2 Frontend — present ✅ (mostly)
- Pages exist: `ProductsList.tsx`, `ProductForm.tsx` (9-tab create/edit), `Categories.tsx` (+`CategoryTree`, `CategoryForm`, `AttributeManager`, `BulkCreatePanel`, `DeleteCategoryModal`), `Brands.tsx` (+`BrandsTable`, `BrandForm`, `DeleteBrandModal`), `ProductsFilterBar.tsx`, plus `VariantManager`, `GalleryManager`, `RelatedProductsManager`, `BulkProductImportModal`.
- Routes (`routes.tsx:164–191`) register products/categories/brands; sidebar gated by `product:read` / `category:*` / `brand:*`.

### 1.3 Gaps & Risks — Catalog CRUD
| # | Severity | Gap | Location | Recommendation |
|---|----------|-----|----------|----------------|
| C1 | Low | **No standalone `Attributes` route/page.** Attributes only editable inline inside Categories. The plan (`frontend-admin-plan.md:55,64`) lists `Attributes` as a first-class page. | `routes.tsx`, missing `src/admin/features/catalog/Attributes.tsx` | Add `/catalog/attributes` route + page, or document the inline-only design as intended. |

## 2. Notifications

### 2.1 Backend — present ✅ (infrastructure), ❌ (catalog)
- **Infra**: `notification.model.js`, `notification-setting.model.js`, `notification-template.model.js`; `notification.service.js` (`sendToUser` → Redis queue `notification-queue.service.js`, worker every 5s, retries + DLQ); email (EmailService) + SMS (Twilio) fan-out; `notification_templates`/`notification_settings`/`notifications`/`sms_logs`/`email_logs` tables (`sql/016`).
- **User endpoints** (`notification.routes.js`): `/notifications/`, `/unread-count`, `/:id/read`, `/mark-all-read`, `/:id` (DELETE), `/settings`.
- **Admin endpoints** (`admin/notification.routes.js`): `/admin/notifications/`, `/send`, `/broadcast`, `/test-email`, `/health`, `/templates` (+ CRUD).
- **Triggered by**: orders, payments, checkout, auth, registration, invitations. **Not catalog.**

### 2.2 Frontend — present ✅ (center + admin)
- `NotificationBell.tsx` + `useNotifications.ts` (polls `/unread-count` every 30s, marks read/dismisses). `SendBroadcast.tsx` (`POST /admin/notifications/broadcast`), `Templates.tsx` (template CRUD). No catalog-notification UI.

### 2.3 Gaps & Risks — Notifications
| # | Severity | Gap | Location | Recommendation |
|---|----------|-----|----------|----------------|
| N1 | **High** | **No catalog events trigger notifications.** Product/category/brand create/update/delete do not call `NotificationService`. No `product.created`/`category.updated`/`brand.deleted` templates. | `product*.controller.js`, `category/brand controllers`, `sql/027` templates | If catalog notifications are in scope, add `sendToUser` calls (+ templates) on catalog mutations (e.g., notify store owner on new product, low-stock). |
| N2 | **High** | **Low-stock alerting is defined but unwired.** `low_stock` template seeded (`sql/016:7`), `inventory_alerts` config + `configureAlert` exist, but **nothing scans stock or sends the alert** (`product.model.js:177` only flips status to `out_of_stock`). | `inventory.service.js`, `product.model.js` | Implement a stock-threshold watcher (scheduled job or on-adjust hook) that calls `sendToUser('low_stock', …)` to `notify_emails`. This is the most likely "catalog notification" the business wants. |
| N3 | Medium | **`/admin/notifications/broadcast` does not fan out email/SMS.** It creates one `user_id=NULL` in-app row; email/SMS broadcast is explicitly stubbed. | `notification.service.js:120–143` | Implement audience fan-out (ALL/ACTIVE customers, staff) across email+SMS, or document broadcast as in-app-only. |
| N4 | Medium | **`POST /admin/notifications/send` bypasses templates, user settings, and the queue** (raw `NotificationModel.create` insert). Inconsistent with the templated path. | `admin/notification.admin.controller.js:23` | Route through `NotificationService.sendToUser` for channel/setting/queue consistency. |
| N5 | Medium | **No real-time delivery.** No websockets/SSE; in-app notifications rely on 30s polling. Acceptable but not "live". | (absent) | Add SSE/WebSocket push if real-time is a requirement; otherwise document polling as intended. |
| N6 | Low | **No frontend view of catalog/low-stock notifications.** Since backend doesn't emit them (N1/N2), the bell never shows catalog events. | `useNotifications.ts`, `NotificationBell.tsx` | Wire after N1/N2; ensure bell renders `type` (low_stock etc.) with deep-links to the affected product. |
| N7 | Low | Catalog `product:write` mutations don't notify the **acting admin's own session** either (e.g., "your product was updated by X"). | — | Optional: emit in-app audit-adjacent notification on catalog change. |

## 3. Audit Log

### 3.1 Backend — present ✅ (capture), ⚠️ (robustness)
- **Schema**: `audit_logs` (`sql/020`): `user_id, action, resource_type, resource_id, old_values JSONB, new_values JSONB, ip_address, user_agent, request_id, created_at`.
- **Logging**: `AuditService` (`audit.service.js`) + `AuditLogModel`. **Catalog IS audited**: products (create/bulk/update/delete + image/variant/related), categories (create/bulk/update/delete/reorder), brands (create/update/delete), attributes (create/update/delete), inventory. Actions: `product.created`, `category.updated` (with old/new diffs), `brand.deleted`, etc.
- **Admin endpoints** (`admin/audit.routes.js` + `admin/audit.controller.js`): `GET /admin/audit` (general `findAll`), `/audit/auth` (`user.login.%`), `/audit/admin-auth` (`user.admin_login.%`), `/audit/stats` (hard-coded login counts).

### 3.2 Frontend — present ✅ (page), ❌ (rendering bugs)
- `AuditLogs.tsx`: tabs System Activity / Admin Logins / Customer Logins; calls `/admin/audit`, `/admin/audit/admin-auth`, `/admin/audit/auth`. **Does NOT call `/admin/audit/stats`.**

### 3.3 Gaps & Risks — Audit Log
| # | Severity | Gap | Location | Recommendation |
|---|----------|-----|----------|----------------|
| A1 | **High** | **Audit UI references non-existent fields.** Activity columns use `entity_type`/`entity_id`, but the API returns `resource_type`/`resource_id`. Catalog entries (e.g. `product.created`) appear but the **Entity column renders blank**. | `AuditLogs.tsx:47–54` | Rename to `resource_type`/`resource_id`. Verify API shape in `audit-log.model.js` (`findAdminEvents`/`findAll`). |
| A2 | **High** | **Auth tabs query the wrong shape.** `/audit/auth` and `/audit/admin-auth` hit `audit_logs` (columns: `user_id, action, ip_address, user_agent…`), but the UI renders `email`, `event_type`, `status` — none of which exist in `audit_logs`. Those columns render blank/undefined. | `AuditLogs.tsx:56–86`, `audit.controller.js:31–63` | Either (a) join `users` to return `email`, or (b) change UI to `user_id`/`action`/`ip_address`. Decide whether auth logs belong in `audit_logs` or the separate `admin_auth_logs` table. |
| A3 | Medium | **No catalog filter / resource drill-down in Audit UI.** Activity tab has no filter for `resource_type=product|category|brand` or `action` prefix, so catalog changes are buried in all system events. | `AuditLogs.tsx` | Add `resourceType`/`action` filter controls (the API already supports them). |
| A4 | Medium | **No diff viewer.** `old_values`/`new_values` exist in DB (captured for catalog updates) but the UI shows only action + entity — admins can't see *what changed*. | `AuditLogs.tsx` | Add expandable row showing old↔new JSON/field diff. |
| A5 | Medium | **`/admin/audit/stats` unused.** Stats endpoint exists (login/lockout counts) but no UI consumes it. | `AuditLogs.tsx`, `audit.controller.js:69` | Surface a stats panel, or extend stats to catalog events (products created/deleted counts). |
| A6 | Medium | **Audit failures are silently swallowed.** `AuditLogModel.log` only `console.error`s on failure and never throws, so a logging outage loses the trail with no alerting. | `audit-log.model.js:17–20` | Add dead-letter/metric on failure; consider failing loudly for security-critical events. |
| A7 | Low | **Diffs capture only a subset of fields** (e.g., product old/new = `{name, sku, status}`), not full-row before/after. Most non-catalog events pass `null` diffs. | `product.controller.js:78–81`, others | Capture fuller diffs (or full snapshots) where change review matters. |
| A8 | Low | Catalog `action` values (`product.created`) are **not** `admin.`-prefixed, so they appear under generic "System Activity" only — fine, but there's no `catalog` action-prefix convenience (the model's `findAdminEvents` defaults to `admin.`). | `audit-log.model.js:34` | Add a `resourceType: 'product'` convenience or document that catalog is under the general endpoint. |
