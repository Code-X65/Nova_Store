# Nova Store Backend — Pre-Deployment Checklist

---

## 🔴 CRITICAL — Must Fix Before Any Production Launch

These are hard blockers. Deploying with them unaddressed causes data integrity loss, financial harm, or security compromise.

| # | Item | File(s) | Risk |
|---|------|---------|------|
| 1 | **SMS service is pure stub** — returns fake success for every SMS notification silently. Order updates and password-reset SMS are never delivered. | `src/services/sms.service.js` | Silent failures in user comms; password-reset broken if SMS is chosen |
| 2 | **Coupon usage never recorded** — `validateAndApplyCoupon` calculates the discount but never calls `incrementUsage()` or `logUserUsage()`. Single-use coupons can be reused infinitely. | `src/services/coupon.service.js` | Direct revenue loss from coupon abuse |
| 3 | **RBAC inconsistency in auth routes** — `auth.routes.js` uses `authorize('ADMIN')` checking legacy `req.user.role`, while all other admin routes use `hasPermission()` / `req.user.roles`. A user with RBAC-issued permissions but `role !== 'ADMIN'` is denied by auth routes but accepted elsewhere. Two authorization paradigms co-exist. | `src/routes/auth.routes.js:224-243`, `src/middlewares/role.middleware.js` | Privilege-escalation / authorization bypass |
| 4 | **Maintenance bypass allows user registration** — All `/api/v1/auth` routes (including `/register`) pass the maintenance check. Bad actors can create accounts during a maintenance window. | `app.js:50-51` (global middleware scope) | Account abuse during declared downtime |
| 5 | **Stripe webhook signature never verified** — `verifySignature` in `payment.service.js` returns `true` unconditionally. Any third party can POST fake payment events to the webhook endpoint. | `src/services/payment.service.js` | Fake payment events processed; orders fulfilled without real payment |

## 🟠 HIGH PRIORITY — Fix Before or Within First Sprint Post-Launch

These are production-quality gaps. Some cause functional defects, others reduce operational observability.

| # | Item | File(s) | Risk |
|---|------|---------|------|
| 6 | **No refund/return endpoints** — DB schema supports `refunded`/`return_status` but no API exists. Customer refund requests can't be fulfilled. | `src/services/order.service.js`, `src/controllers/order.controller.js` | Broken customer support workflow |
| 7 | **Audit logger exists unused** — `audit.service.js` is fully implemented but never called from any business service or controller. No audit trail for critical operations. | `src/services/audit.service.js` | Compliance/regulatory gap; can't trace who changed what |
| 8 | **No session/token garbage collection** — Expired sessions in DB, expired/used verification and reset tokens in DB and Redis accumulate indefinitely. | `src/models/session.model.js`, `src/models/token.model.js` | DB bloat; performance degradation over time |
| 9 | **Inventory oversell on concurrent checkout** — No reservation step before payment. Two simultaneous checkouts for the same product can both "succeed" on stock, overselling the item. | `src/services/inventory.service.js`, `src/services/checkout.service.js` | Sell items you don't have; customer dissatisfaction |

## 🚀 Recommended Deployment Strategy

1. **Fix all 🔴 blockers first** (items 1–5). Any one of them opens the business to measurable financial or security damage on day one.
2. **Fix 🟠 items 6–9.** These are also needed for a functional go-live (refunds, audit trail, inventory integrity).
3. **Ship immediately after steps 1 & 2.** Do not wait for medium/low items.
4. **Address 🟡 items in Sprint 1** post-launch (1–2 week window).
5. **Address 🟢 items in Sprint 2–3** post-launch or as time permits.

- Cart + checkout + payment (with real Stripe verification) ✅
