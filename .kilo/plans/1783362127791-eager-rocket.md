# Nova Store — Admin Frontend Plan (Store Owner, Manager, Inventory Staff, Order Staff)

> Builds on the backend readiness plan. **Pre-condition reminder:** RBAC migrations `sql/053–057` must be applied to the live DB (`npm run db:migrate` + `verify-store-migration.js`) before the admin UI depends on staff roles / `store_id`. See that plan's B3.

## 1. Stack & Decisions (confirmed with user)
- **React + Vite + TypeScript + Tailwind CSS** (CORS already allows `http://localhost:5173`).
- **Admin panels only** (no customer storefront in this plan).
- **Single SPA with a shared shell**; role-based sidebar nav derived from permissions returned by `/api/v1/admin/my-permissions`.

### Supporting libraries
- **React Router v6** — routing + protected routes.
- **TanStack Query (React Query)** — data fetching, caching, invalidation.
- **Axios** with interceptors — session-cookie auth + CSRF header injection.
- **React Hook Form + Zod** — forms & validation.
- **TanStack Table** — sortable/filterable data tables.
- **Recharts** — analytics/dashboard charts.
- **Tailwind + Headless UI** — modals, menus, toasts.
- **React Query + a small `usePagination`/`useFilters` helper** for list pages.

## 3. Auth & API Client Layer (critical — admin uses SESSION COOKIES + CSRF)
- **Login** `POST /api/v1/admin/login` → sets **session cookie** (not just JWT). Frontend must send requests with `withCredentials: true`.
- **CSRF**: before any state-changing request (POST/PUT/PATCH/DELETE), `GET /api/v1/auth/csrf-token` returns a token; send it as header `x-csrf-token`. Axios interceptor: on app load (after login/verify) fetch CSRF once and attach to mutations. (GET/HEAD/OPTIONS are exempt; Bearer-JWT requests are exempt.)
- **Session check**: on app boot call `GET /api/v1/admin/verify`. 401 → redirect to `/admin/login`.
- **Permissions**: call `GET /api/v1/admin/my-permissions` once per session → store `{ roles, permissions }` to drive nav + button-level gating (`hasPermission('order:write')`).
- **Logout** `POST /api/v1/admin/logout` (clears cookie).
- 401 interceptor → clear session, redirect to login.

## 4. Shared Shell (every role)
- **Login Page** (`/admin/login`): email + password, rate-limited server-side. On success → verify + my-permissions → redirect to role dashboard.
- **AppShell**: left **Sidebar** (menu items filtered by permissions), top **Topbar** with **Store name** (from `storeContext`), **NotificationBell**, and **UserMenu** (profile, active sessions, logout).
- **NotificationBell** (all roles) — user-facing notification center:
  - On mount + poll (30s): `GET /api/v1/notifications/unread-count` → red badge.
  - Dropdown: `GET /api/v1/notifications/` lists items (order updates, inventory alerts, invites).
  - `PUT /api/v1/notifications/:id/read` (click item), `POST /api/v1/notifications/mark-all-read`.
  - `DELETE /api/v1/notifications/:id` (dismiss).
- **Sessions** (UserMenu → "Active sessions"): `GET /api/v1/admin/sessions/`, `DELETE /api/v1/admin/sessions/:sessionId`, `DELETE /api/v1/admin/sessions/` (revoke all).

## 5. Page Inventory by Module
Each module lists **what the page entails** and **endpoints**. Roles that see the module are noted; "read-only" means list/view only.

### 5.1 Dashboard / Overview (home, role-specific)
- **STORE_OWNER & MANAGER — Full analytics dashboard**: StatCards (revenue, orders, users, low-stock), revenue trend chart, best-sellers, order-status breakdown, payment mix.
  - `GET /admin/analytics/dashboard`, `/analytics/revenue`, `/revenue/summary`, `/best-sellers`, `/users`, `/orders`, `/payments`, `/inventory`, `/sales/reports`, `/sales/daily-summary`, `/dashboard/order-stats`, `/dashboard/inventory-stats`.
- **ORDER_STAFF — Fulfillment dashboard**: order KPI cards, dispatch queue preview, recent orders needing action.
  - `GET /admin/dashboard/order-stats`, `GET /orders/admin/dispatch-queue`, `GET /orders/admin/list`.
- **INVENTORY_STAFF — Inventory dashboard**: stock-health cards, low-stock preview, sales summary, top products.
  - `GET /admin/dashboard/inventory-stats`, `GET /inventory/low-stock`, `GET /admin/sales/daily-summary`, `/admin/sales/top-products`.

### 5.2 Orders (ORDER_STAFF full · MANAGER incl. returns · STORE_OWNER all · INVENTORY_STAFF read-only)
- **Orders List**: filterable/sortable table (status, date, customer, total, payment). Export button.
  - `GET /orders/admin/list`, `GET /orders/admin/export` (CSV).
- **Order Detail**: customer info, items, address, payment, status timeline, **fulfillment action buttons**:
  - `POST /orders/admin/:id/ready`, `/dispatch`, `/picked-up`, `/out-for-delivery`, `/delivery-attempted`, `/deliver` (POD), `/returned-to-store`, `/ship`; generic `PATCH /orders/admin/:id`; `POST /orders/admin/bulk-action`.
  - Gated by `requireOrderStaff` (ORDER_STAFF/MANAGER/OWNER). Returns (`POST /orders/admin/:id/return`) gated by `requireManager`.
- **Dispatch Queue**: dedicated board of ready-to-dispatch orders. `GET /orders/admin/dispatch-queue`.
- **Returns** (MANAGER+): `POST /orders/admin/:id/return`.

### 5.3 Inventory (INVENTORY_STAFF full · MANAGER · STORE_OWNER · not ORDER_STAFF)
- **Stock Levels**: per-product stock table with search/filter. `GET /inventory/:id` (detail), `GET /inventory/transactions` (history).
- **Adjust Stock**: add/reduce forms. `POST /inventory/stock`, `POST /inventory/reduce`, `POST /inventory/bulk-update`.
- **Low Stock**: alert list. `GET /inventory/low-stock`.
- **Thresholds & Alerts**: set reorder threshold `PUT /inventory/:id/threshold`; configure alerts `GET /inventory/alerts`, `POST /inventory/alerts`.

### 5.4 Catalog — Products / Categories / Brands / Attributes (MANAGER · STORE_OWNER; INVENTORY_STAFF read-only via `product:read`)
- **Products List + Create/Edit**: table, image gallery upload (`POST /products/:id/images`, `DELETE .../images/:index`), variants (`PUT/DELETE /products/:id/variants/:variantId`).
  - `GET /products` (public list for selectors), `POST /products` (create), `PATCH /products/:id`, `DELETE /products/:id`.
- **Categories**: tree, create/bulk, edit, delete, attributes.
  - `POST /categories`, `/categories/bulk`, `PATCH /categories/:id`, `DELETE /categories/:id`, `POST /categories/:id/attributes`, `PUT/DELETE /attributes/:attributeId`.
- **Brands**: `POST /brands`, `PATCH /brands/:id`, `DELETE /brands/:id`.
- **Attributes** (category): managed via category + `/attributes` endpoints above.

### 5.5 Customers / Users (MANAGER · STORE_OWNER)
- **Users List**: `GET /admin/users/` (user:read). **User Detail / status**: `PATCH /admin/users/:id` (user:write).

### 5.6 Coupons (MANAGER · STORE_OWNER)
- **Coupons List + Create/Edit + Bulk-generate + Deactivate + Usage analytics**:
  - `GET /admin/coupons`, `POST /admin/coupons`, `/bulk-generate`, `GET /admin/coupons/:id`, `PATCH /admin/coupons/:id`, `DELETE /admin/coupons/:id`, `POST /admin/coupons/:id/deactivate`, `GET /admin/coupons/:id/usage`.

### 5.7 Shipping (MANAGER · STORE_OWNER)
- **Zones & Rates CRUD**: `GET/POST/PUT/DELETE /admin/shipping/zones[/:id]`, `GET/POST/PUT/DELETE /admin/shipping/rates[/:id]`.

### 5.8 Reviews & Reports (MANAGER · STORE_OWNER)
- **Reviews Moderation**: `GET /admin/reviews`, `PATCH /admin/reviews/:id`, `DELETE /admin/reviews/:id`, `POST /admin/reviews/bulk-action`.
- **Review Reports Queue**: `GET /admin/review-reports`, `PATCH /admin/review-reports/:id`, `GET /admin/review-reports/summary`.

### 5.9 Staff & Roles (MANAGER · STORE_OWNER; owner-only: revoke admin)
- **Invitations**: create (role + store), list, resend, revoke. `POST/GET/DELETE /admin/invitations[/:id]`, `POST /admin/invitations/:id/resend`.
- **Staff List / Detail**: `GET /admin/`, `GET /admin/:id`, `GET /admin/:id/permissions`.
- **Role/Permission Management**: `PATCH /admin/:id/roles`, `PATCH /admin/:id/permissions` (MANAGER); `DELETE /admin/:id` (STORE_OWNER only).
- **Roles & Permissions directory**: `GET /roles`, `GET /roles/:id`, `POST /roles`, `PATCH/DELETE /roles/:id`, `POST /roles/:id/permissions`, `GET /permissions`, `/permissions/categories`, `GET /permissions/:id`, `GET/POST/DELETE /user-routes/:userId[/:roleId]`.
- **My Permissions** page: `GET /admin/my-permissions` (self-service view of own access).

### 5.10 Sales & Analytics — cross-department read (ALL FOUR ROLES; owner/manager full, staff read-only)
- **Sales Reports**: `GET /admin/sales/reports`, `/sales/daily-summary`, `/sales/top-products`, `/sales/order-tracking` (all use `sales:read`/`order:read` which staff possess).
- ORDER_STAFF & INVENTORY_STAFF get read-only views here (no write controls).

### 5.11 Store Settings (MANAGER · STORE_OWNER)
- **General + Store Settings + Currencies + Test Email**:
  - `GET /admin/settings`, `PATCH /admin/settings/bulk`, `GET/PUT /admin/settings/:key`, `GET /admin/settings/history/:key`, `POST /admin/settings/test-email`, `PUT /currencies/admin/:code`.

### 5.12 Audit & Security (STORE_OWNER by default; MANAGER needs `audit:read` — see §7)
- **Audit Logs**: `GET /admin/audit`, `/audit/auth`, `/audit/admin-auth`, `/audit/stats`.
- **Sessions** (all admins): `GET /admin/sessions`, `/sessions/health`, revoke endpoints.
- **Migrations**: `GET /admin/migrations/status`.

### 5.13 Notifications Admin (STORE_OWNER · MANAGER)
- **Send / Broadcast / Templates / Health**: `GET /admin/notifications`, `POST /admin/notifications/send`, `/broadcast`, `/test-email`, `GET /admin/notifications/health`, `GET/POST/PUT/DELETE /admin/notifications/templates[/:id]`.

## 7. Backend Permission Gaps to Resolve (BEFORE/with frontend)
The frontend assumes MANAGER can reach Catalog, Settings, Audit, Analytics, Shipping, Reviews. But the **default MANAGER seed lacks** `category:create/write`, `brand:create/write`, `analytics:read`, `settings:read/write`, `shipping:write`, `review:write`, `audit:read` — and the route guards use those exact keys. Consequences:
- MANAGER blocked from Categories/Brands (guard uses `category:create/write`, `brand:create/write`, not the seeded `category:manage`/`brand:manage`).
- MANAGER blocked from Settings, Audit, Analytics dashboards, Shipping, Reviews.

**Fix options (pick one, apply in `sql/057` or a new migration):**
1. Add the missing permission keys to the MANAGER seed (`role_permissions`), OR
2. Change route guards to the `*:manage` keys the seed already has.

Also confirm ORDER_STAFF/INVENTORY_STAFF can reach the cross-dept Sales endpoints (they have `sales:read`/`order:read` — OK) and their dashboard stats (`order:read` / `inventory:read` — OK).

## 9. Notes for Implementation
- All admin routes sit under `/api/v1/admin/*` with global `requireAdmin` + `storeContext` + `scopeToStore`; frontend just needs a valid admin session — store scoping is server-enforced.
- Public read endpoints (`GET /products`, `/categories`, `/brands`) are used inside admin forms (product/category/brand selectors).
- File uploads (product images, admin docs) use `multipart/form-data` to `POST /products/:id/images` and `POST /admin/upload`.
- Charts: Recharts fed by `/analytics/*` and `/sales/*` JSON.
- Export endpoints return CSV downloads.
