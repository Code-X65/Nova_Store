# Backend Readiness Assessment — "What's Left Before Frontend"

The backend is **substantially complete and API-ready** (~90%+). Every domain a frontend would need — auth, products, cart, wishlist, checkout, payments, orders, inventory, reviews, coupons, shipping, notifications, RBAC/admin, analytics/sales, stores — has **fully implemented routes → controllers → services → models** with no stubs. The remaining open item is **B3: confirm the RBAC/store migrations (053–057) have actually been applied to the live database** (the SQL + code is present and committed, but the DB execution is unverified from the repo alone).

### B2. `checkout.controller.js` store-scoping bug

`src/controllers/checkout.controller.js:14,28` calls `CartService.getOrCreateCart(userId, sessionId, cartId, req.store?.id)`.
`CartService.getOrCreateCart(userId, sessionId, storeId)` only takes **3 args**, so the `cartId` is being passed into the **`storeId`** position and `req.store?.id` is silently dropped.
**Fix:** drop the `cartId` arg (or correct the call order) so store context is preserved.

### B3. RBAC migrations (053–057) are UNCOMMITTED and UNAPPLIED

The `stores` table, `store_id` columns, store-staff roles (`STORE_OWNER/MANAGER/ORDER_STAFF/INVENTORY_STAFF`), and store-aware RPCs exist only as files in `sql/` and as uncommitted code (`git status` shows `?? sql/053..057`, `?? src/middlewares/*`, `?? src/models/store.model.js`, etc.). They have **not been run against the live Supabase DB**.
**Fix:** run `npm run db:migrate` (or `node src/scripts/verify-store-migration.js`) and confirm before the admin/frontend depends on `store_id` / staff roles. Until then, admin auth relying on the new role model may fail.

**Reliable API contract surfaces for the frontend:**

- `req.user` (bearer/JWT customer) and `req.admin` (session-cookie admin) both carry `{ id, role, roles, store_id, permissions }` plus a `hasRole(...)` helper on `req.admin`.
- Frontend should read role/permission state from an endpoint (e.g. `/api/v1/admin/my-permissions`, `/api/v1/auth/oauth/status`, `/api/v1/user/profile`), NOT rely on serialized `req.admin`.
- Guest flows use `optionalAuth` + `x-session-id` header; admin flows need `GET /api/v1/auth/csrf-token` + CSRF header.
