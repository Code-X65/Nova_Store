# Comprehensive Gap Analysis — Inventory Management System

## 1. BACKEND ARCHITECTURE

### 1.1 Completed Features
| # | Feature | Status |
|---|---------|--------|
| 1.1 | **Core Stock Operations** — Add stock, reduce stock, manual adjustment, bulk update | Fully implemented via `inventory.service.js` + controller + RPCs |
| 1.2 | **Variant-Level Inventory** — `product_variants.stock_quantity` tracked; `updateStock()` handles variant atomic updates | Fully implemented |
| 1.3 | **Transaction Audit Trail** — `inventory_transactions` table with paginated history, store-scoped | Fully implemented |
| 1.4 | **Low-Stock Detection** — `GET /inventory/low-stock` using `get_low_stock_products` RPC + fallback | Fully implemented |
| 1.5 | **Threshold Management** — `PUT /inventory/:id/threshold` + `inventory_alerts` config table | Fully implemented |
| 1.6 | **Stock Reservation System** — Atomic RPCs for checkout holds, commit on payment, release on expiry | Fully implemented (`reserve_stock_increment`, `release_stock_reservation`, `commit_reserved_stock`, `release_session_reservations`, `release_expired_reservations`) |
| 1.7 | **Return/Cancellation Restoration** — Order cancellation and sellable returns restore stock via `addStock()` | Fully implemented |
| 1.8 | **RBAC & Middleware** — `inventory:read`, `inventory:write`, `inventory:alert` permissions enforced per route | Fully implemented |
| 1.9 | **Event Bus & Alerts** — `inventory.low_stock`, `inventory.out_of_stock`, `inventory.discrepancy` events emitted; notification routing configured | Fully implemented (events emitted) |
| 1.10 | **Database Schema** — `products`, `product_variants`, `inventory_transactions`, `inventory_alerts`, `inventory_reservations` with indexes and store-scoping | Fully implemented |
| 1.11 | **Reservation Cleanup Job** — `reservation-cleanup.job.js` + `release_expired_reservations` RPC | Implemented but **not wired to a scheduler** |

| 2.6 | **Negative Stock Prevention** | No atomic pre-check before `updateStock()` | `quantityChange` not validated against available stock at DB level |

## 3. CROSS-CUTTING GAPS (Backend ↔ Frontend)

| ID | Gap | Impact |
|----|-----|--------|
| X1 | **No shared type contract enforcement** — backend returns snake_case, frontend types are camelCase and unused | Silent data loss |
| X2 | **Cart stock validation missing on add** — backend only validates at checkout | Users add unavailable items to cart |
| X3 | **No warehouse/multi-location support** on either backend or frontend | Single-store limitation blocks multi-warehouse operators |
| X4 | **No stock transfer / purchase order / supplier reorder workflow** on either side | Manual-only restocking; no procurement automation |
| X5 | **Zero end-to-end tests for inventory flows** on both backend and frontend | High regression risk for future changes |

---

## 4. EXECUTION NOTES
- Backend gap remediation should prioritize test coverage (B-H1) before schema changes.
- Frontend gap remediation should start with the Alerts CRUD fix (F-H1) and navigation (F-H4) as they have immediate user impact.
- Cross-cutting type contract (X1) should be addressed alongside F-H3 to prevent schema drift.
