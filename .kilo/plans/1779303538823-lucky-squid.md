## 1. Current State Analysis

### Registration Flow (Today)
```
POST /register
  Body: { email, password, firstName, lastName }
  → Creates user with is_email_verified=false
  → Sends verification email (verification_tokens)
  → Returns 201
```

**What's missing:**
- No `confirmPassword` validation on registration (frontend-only currently)
- No `phone_number` OTP verification flow
- No `home_address` captured during registration
- No `referral_source` captured during registration
- No `referred_by` referral link/code tracking
- No `is_phone_verified` flag on users

## 2. SchemaChanges

### A. `sql/029_phone_verification_tokens.sql` *(new)*
Creates phone OTP share table same pattern as `password_resets` / `email_verification_tokens`.

```sql
CREATE TABLE IF NOT EXISTS phone_verification_tokens (
  id            UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token         TEXT NOT NULL,         -- 6-digit OTP
  phone_number  TEXT NOT NULL,         -- number the OTP was sent to
  expires_at    TIMESTAMP WITH TIME ZONE NOT NULL,
  used          BOOLEAN DEFAULT FALSE,
  attempt_count INT DEFAULT 0,         -- track brute-force attempts
  created_at    TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_phone_verification_token_user    ON phone_verification_tokens(user_id);
CREATE INDEX IF NOT EXISTS idx_phone_verification_token_phone   ON phone_verification_tokens(phone_number);
CREATE INDEX IF NOT EXISTS idx_phone_verification_token_token    ON phone_verification_tokens(token);
```

### B. `sql/030_registration_fields.sql` *(new)*
Adds four columns to the `users` table + seed notification templates.

```sql
-- 1. Home address (compact JSON: { street, city, state, postalCode, country })
ALTER TABLE users ADD COLUMN IF NOT EXISTS home_address JSONB;

-- 2. Referral-source tracking at signup
ALTER TABLE users ADD COLUMN IF NOT EXISTS referral_source TEXT;   -- e.g. 'facebook', 'instagram', 'friend'
ALTER TABLE users ADD COLUMN IF NOT EXISTS referred_byUUID;      -- user.id of referrer

-- 3. Phone verification status
ALTER TABLE users ADD COLUMN IF NOT EXISTS is_phone_verified BOOLEAN DEFAULT FALSE;

-- 4. Indexes
CREATE INDEX IF NOT EXISTS idx_users_referred_by     ON users(referred_by);
CREATE INDEX IF NOT EXISTS idx_users_referral_source ON users(referral_source);
```

## 3. API Endpoints

### 3.1 Enhanced `POST /api/v1/auth/register`

**Complete replaces existing register. Collects full form:**

```json
{
  "email":    "user@example.com",
  "password": "StrongPass123!",
  "confirmPassword": "StrongPass123!",
  "firstName": "John",
  "lastName":  "Doe",
  "phoneNumber": "+234801234567",
  "phoneCountryCode": "+234",           // prefill for Nigerian user base
  "homeAddress": {
    "street": "123 Main Street",
    "city":   "Lagos",
    "state":  "Lagos",
    "postalCode": "100001",
    "country": "Nigeria"
  },
  "referralSource": "facebook",         // optional — "facebook", "instagram", "friend", "other"
  "referredByCode": "NOVA-XYZ123"       // optional — 12-char referral code (resolved to user.id)
}
```

**Behavior:**
- Server-side `password === confirmPassword` check (400 if mismatch)
- Creates user with:
  - `password_hash` (bcrypt 12 rounds)
  - `phone_number`, `phone_country_code`, `home_address`
  - `is_phone_verified = false`
  - `referral_source`, `referred_by` (resolved from code if provided)
- Sends **phone OTP** via Twilio SMS (6-digit code, 10-min expiry)
- Sends **email verification** email (1 received, clashes`)
- Sends **referral credit** email to referrer (if `referred_by` present)
- Returns `userId` and `messageId` back in 201 response (frontend uses `userId` to submit OTP verify)

**Swagger update** — update existing `register` Swagger block with all new fields.

---

### 3.2 `POST /api/v1/auth/send-phone-otp`

Resend or churn phone OTP to any number. Rate-limited.

```json
{ "userId": "uuid", "phoneNumber": "+234801234567", "phoneCountryCode": "+234" }
```

**Response:**
```json
{ "success": true, "message": "OTP sent to +234 801 234 567" }
```

---

### 3.3 `POST /api/v1/auth/verify-phone`

Verify the 6-digit OTP.

```json
{ "userId": "uuid", "otp": "123456" }
```

**Behavior:**
- Looks up `phone_verification_tokens` for matching user + token + `expires_at > now()`
- Checks `attempt_count` — reject after 5 failed attempts (require `send-phone-otp`)
- On success: sets `users.is_phone_verified = true`, marks token `used = true`
- Returns remaining unverified fields
- On failure: increments `attempt_count`, returns 400 with remaining attempts

## 5. Service Layer (`registration.service.js`) — Full Method List

```
RegistrationService
├── register(body)              — main: validate, create user, send emails + SMS OTP
├── validatePassword(body)      — confirms password === password
├── resolveReferralCode(code)   — find user by referral code, return user.id or null (handle: self-referral, expired users)
├── sendPhoneOTP(userId, phone) — generate 6-digit OTP, send via Twilio, store in phone_verification_tokens
├── verifyPhoneOTP(userId, code) — validate OTP, mark verified, clean up
├── handleReferralCredit(toUserId, fromUserId)  — grant loyalty_points / notify referrer
└── resendPhoneOTP(userId)      — invalidate old, generate new, re-send
```

**Password validation rules** (same as login): min 12 chars, upper+lower+number+special char. `password === confirmPassword` is an additional server-side cross-check.

**Phone OTP format:**
- 6-digit numeric code
- Expiry: 10 minutes
- Max failed attempts: 5 (lock then redistribution)
- Stored hashed only (SHA-256)

**SMS template:**
```
Your Nova Store verification code is: 123456.
Valid for 10 minutes. Do not share this code.
```

## 8. Data Flow Diagram

```
Frontend (multi-step registration form)
  │
  ├─ Step 1: Email + Password + Confirm Password
  │     Frontend: inline confirmPassword match check
  │     POST /register → Creates user record (unverified, API returns 201 with userId)
  │     ← sends email verif + Twilio SMS OTP + (optional) credit referrer
  │
  ├─ Step 2: Email Verification (existing flow — redirect link)
  │     GET /verify-email?token=xxx → is_email_verified = true
  │
  ├─ Step 3: Phone OTP Verification
  │     POST /verify-phone { userId, otp }
  │     ← sets is_phone_verified = true
  │
  └─ Step 4: Optional: Home Address + Referral Source
        PUT /profile { homeAddress, referralSource, referredByCode }
        → updates existing user record
```

**Note:** Steps 2 and 3 run in parallel — frontend can verify both simultaneously. Step 4 is optional.

## 9. Self-Referral Protection

When `referred_by_code` is supplied during registration:

1. Look up the code → `referrer_user_id`
2. If `referrer_user_id === new_user.id` → reject with 400 (can't refer yourself)
3. If referrer user does not exist / code expired → silently ignore, do not fail registration
4. If referrer user is `is_active = false` or `deleted_at IS NOT NULL` → silently ignore
5. Else → store `users.referred_by = referrer_user.id` + `users.referral_source = 'code'`

## 10. Validation Red Flag Rules (Joi schema additions)

```javascript
confirmPassword: Joi.string().required()
  .valid(Joi.ref('password'))   // same as password
  .messages({ 'any.only': 'Passwords do not match' }),

phoneNumber:  Joi.string().pattern(/^\+?\d{7,15}$/).required(),
phoneCountryCode: Joi.string().pattern(/^\+?\d{1,4}$/).required(),

homeAddress:  Joi.object({
  street:    Joi.string().max(200).required(),
  city:      Joi.string().max(100).required(),
  state:     Joi.string().max(100).required(),
  postalCode: Joi.string().max(20).required(),
  country:   Joi.string().max(100).required(),
}).required(),

referralSource: Joi.string().valid('facebook','instagram','twitter','tiktok',
  'friend','colleague','google','other').optional(),

referredByCode: Joi.string().alphanum().length(12).optional(),
```

## 13. Error Semantics

| HTTP Code | Condition |
|---|---|
| 201 | Registration successful — OTP sent |
| 400 | Password mismatch, invalid phone format, invalid JSON body |
| 409 | Email already registered |
| 422 | Phone OTP expired / max attempts hit / invalid OTP |
| 429 | Rate limit hit on register / send-phone-otp |
| 500 | SMS provider error (non-critical — user can retry) |

## 14. Edge Cases & Mitigations

| Edge Case | Mitigation |
|---|---|
| Duplicate email during race | DB `UNIQUE` on `users.email` catches; 409 response |
| Duplicate phone_number during race | DB `UNIQUE` on `users.phone_number` catches; 409 response |
| User changes phone mid-OTP-flow | Reject verify-phone if stored number ≠ submitted; require resend |
| Self-referral (`referredByCode === own code`) | Reject registration with 400 + clear message |
| Referrer becomes deleted/inactive after signup | Validated at registration time only; later state doesn't undo referral |
| Twilio unreachable during registration | Return 201 with `warn: "phone_otp_pending"` — frontend shows "retry send" button |
| SMS delivery failure only | Email verification proceeds; registration is not blocked by SMS failure |
| Expired/used OTO resubmission | Find by token + check `used + expires_at`; replace token automatically |
| Massive `home_address` JSON payload | Joi max on individual fields + Express body-parser limit 1MB |
