- Legacy `products.subcategory` TEXT column still exists and is not yet deprecated in `ProductModel` selects.

## Category Attribute Gaps (RESOLVED)

Migration 041 + 042:
- `cleanup_orphaned_product_attributes()` trigger deletes `product_attributes` rows when `category_id` OR `subcategory_id` changes.
- `validate_product_attribute_value()` trigger supports `enum`, `number`, `boolean`, `date` types with strict validation.
- `category_attributes.attribute_type` CHECK includes `date`.
- `product_attributes.attribute_value` **migrated to JSONB** with typed backfill (numbers as JSON numbers, booleans as JSON booleans).
- `get_products_by_attributes` RPC now supports **range queries** on numeric attributes (`>=`, `>`, `<=`, `<`) via JSONB casting.
- 042 one-time cleanup deletes pre-041 orphaned attributes and backfills `subcategory_id`.

### Remaining Risk
- Date attribute values are stored as JSONB strings (not typed); the trigger validates the string format but the column is not natively typed.
- `attributeService.validateAttributes` in code still does not include a `date` case; direct service callers bypassing the trigger (e.g., scripts, tests) could insert invalid dates.
- `get_products_by_attributes` RPC does not support range queries on booleans or dates (only exact match).

### Remaining Risk

- Brand rename path: `sync_product_category_brand_slugs` fetches `name` into `products.brand` on FK change, but the old legacy TEXT column was called `brand` (name) not `slug`. Consistent behaviour masked by column name; verify no code reads `brand.slug` expecting a slug.

## Recommended Fixes (Updated Plan)

### P1 â€” Migrate filters to FK-only, retire TEXT dual-storage
1. Update `ProductModel.findAll` to filter on `category_id` and `brand_id`; leave legacy TEXT columns for display/SEO only.
2. Update `ProductService.createProduct` and `updateProduct` to stop manually writing legacy TEXT slugs; let the trigger handle it.
3. Deprecate `products.category`, `products.subcategory`, and `products.brand` TEXT columns in a future migration once all queries are proven FK-based.

### P2 â€” Expose subcategory via admin API
1. Add admin routes to list available subcategory candidates (children of a chosen parent) within product create/update forms.
2. Update Swagger schemas to include `subcategory_id` and validate parent-child relationship in `productSchema` / `productUpdateSchema` Joi validators.

### P3 â€” Fix slug uniqueness
1. In `ProductService.createProduct`, after generating/appending a slug, query for existence and retry until unique (or use UUID prefix deterministically).
2. In `ProductService.updateProduct`, handle slug collisions if the new slug conflicts with another product.

### P4 â€” Harden app-level attribute validation
1. Add `date` case to `attributeService.validateAttributes`.
2. Consider adding range/prefix support for booleans and dates in `get_products_by_attributes` if product listing pages need it.

### P5 â€” Review RPC / trigger coverage and add integration tests
1. Confirm application-layer code paths exercise the new triggers and RPCs.
2. Add integration tests proving:
   - `get_products_by_attributes` returns correct products for mixed-type filters including numeric ranges.
   - `validate_product_subcategory` throws when child is not a descendant of parent.
   - Brand soft-delete cascades `brand_id` to NULL and prevents listing in `findAll({ activeOnly: true })`.
