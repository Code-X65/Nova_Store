# Comprehensive Gap Analysis: Audit Logging & In-App Notification Systems

**Project:** Nova Store  
**Date:** 2026-07-11  
**Scope:** Full-stack technical audit of backend architecture and frontend implementation  
**Status:** Plan / Research Complete — Ready for Implementation

---

## Executive Summary

The Nova Store backend implements a **well-architected, layered notification and audit system** with:
- `AuditService` for structured audit trail persistence
- `NotificationService` for async queued multi-channel dispatch (email, SMS, in-app)
- `notification-router.service` for event-driven team alerts via SSE
- `audit-context` middleware for automatic actor enrichment
- A rich `audit_logs` schema with delta computation, severity tagging, and export capabilities

Despite this solid foundation, a **full-stack gap analysis reveals 17 critical backend trigger gaps**, **6 metadata/schema inconsistencies**, and **12 frontend UX deficiencies** that prevent the system from achieving 100% coverage, real-time visibility, and human-readable presentation.

## Part 2: Technical Roadmap to 100% Coverage

### Phase 1: Eliminate Dead Code & Standardize Audit Capture (Week 1)

**Goal:** Stop relying on manual `AuditService.log()` calls and enforce consistency.

1. **Deprecate `audit-action.middleware.js` or wire it into at least 3 routes as a proof-of-concept.**
   - Target: `product-category.controller.js` CRUD routes.
   - If successful, roll out to all CRUD controllers.

2. **Create an `AuditGuard` utility that wraps controller methods.**
   ```js
   const guarded = wrapWithAudit(controller.update, { resourceType: 'product', action: 'product.updated' });
   ```
   - Auto-captures `oldValues` from model layer, `newValues` from result, actor from `req.actor`.
   - Guarantees no controller can forget to log.

3. **Fix silent audit failures.**
   - Replace `.catch(() => {})` with `.catch(err => logger.error('[Audit]', err))` in `access.controller.js` and `admin-management.controller.js`.
   - Add an `audit_logs` insert failure counter to `getAuditStats`.

### Phase 2: Backfill Missing Triggers (Week 2)

**Goal:** Add audit logs to all uncovered write endpoints.

| Priority | Endpoint | Action |
|---|---|---|
| P0 | `store.admin.controller.js` — `updateStoreProfile`, `updateStoreSettings` | Add `AuditService.log(req, 'store.updated', 'store', ...)` with old/new values. |
| P0 | `shipping.admin.controller.js` — all 6 CRUD methods | Add audit logs for zones and rates. |
| P0 | `notification.admin.controller.js` — `createTemplate`, `updateTemplate`, `deleteTemplate`, `updateRoutingRule` | Add audit logs with old/new snapshots. |
| P0 | `invitation.controller.js` — `createInvitation`, `revokeInvitation`, `resendInvitation` | Add controller-level audit (service already logs create). |
| P1 | `session.controller.js` — `revokeSession`, `revokeAllSessions` | Add `AuditService.log(req, 'session.revoked', 'session', ...)`. |
| P1 | `rider.controller.js` — `createGuarantor`, `updateGuarantor`, `deleteGuarantor` | Add audit logs for guarantor lifecycle. |
| P1 | `notification.admin.controller.js` — `sendNotification` | Add audit log for ad-hoc sends. |

### Phase 3: Wire Missing Event Emitters & Routing Rules (Week 3)

**Goal:** Ensure every seeded template has a live event emitter and routing rule.

1. **Add event emitters for orphaned events:**
   - `order.service.js:131` — emit `order.cancelled` after `fireNotification`.
   - `order.service.js:230` — emit `order.delivered` after `markDelivered`.
   - `order.service.js:513` — emit `order.returned` after `logHistory(orderId, 'returned', ...)`.
   - `rider.controller.js:223` — emit `rider.approved` after `AuditService.log`.
   - `rider.controller.js:240` — emit `rider.rejected` after `AuditService.log`.

2. **Add missing routing rules:**
   ```sql
   INSERT INTO notification_routing_rules (name, event_key, recipient_roles, channel, severity, template_key)
   VALUES
     ('Order cancelled → Sales', 'order.cancelled', '{ORDER_STAFF,MANAGER}', '{inapp,email}', 'warning', 'order_cancelled'),
     ('Order delivered → Sales', 'order.delivered', '{ORDER_STAFF}', '{inapp}', 'info', 'order_delivered'),
     ('Rider approved → Owners', 'rider.approved', '{STORE_OWNER,MANAGER}', '{inapp}', 'info', 'rider_approved'),
     ('Rider rejected → Owners', 'rider.rejected', '{STORE_OWNER,MANAGER}', '{inapp}', 'warning', 'rider_rejected')
   ON CONFLICT (event_key) DO NOTHING;
   ```

3. **Add `payment.success` template + emitter:**
   - Seed `payment_success` template (email + inapp).
   - Emit `order.payment_succeeded` from `payment.service.js` on successful charge.

### Phase 4: Metadata & Schema Hardening (Week 4)

**Goal:** Ensure every audit entry has complete, queryable metadata.

1. **Standardize `oldValues` capture across all controllers.**
   - Controllers must fetch the pre-change state and pass it to `AuditService.log()`.
   - Enforce via `AuditGuard` wrapper.

2. **Add `notification_dlq` table for compliance.**
   - Persist dead-lettered notification jobs with `job_id`, `reason`, `payload`, `created_at`.
   - `notification-queue.service.js` writes to this table on final failure.

3. **Add `notification_send_audit` join table.**
   - Track every `sendToUser` call with `notification_id`, `template_key`, `channel`, `status`, `sent_at`.
   - Currently only `email_logs` and `sms_logs` exist; unify under one `notification_deliveries` table.

### Phase 5: Frontend Real-Time & UX Overhaul (Week 5-6)

See **Part 3** for detailed UI/UX recommendations. Implementation items:

- Replace native `confirm()` with modal confirms.
- Consolidate SSE connections into a single `RealtimeProvider`.
- Build full Notifications Inbox page.
- Add date-range filtering to AuditLogs.

## Part 3: UI/UX Recommendations

### 3.1 Audit Trail — From Raw Logs to Human-Readable Story

**Current State:**
- `AuditLogs.tsx` renders a TanStack Table with severity pills and raw `JSON.stringify(before → after)` delta expansion.
- No date-range filter; no row-click detail drawer; no auto-refresh.

**Recommendations:**

| # | Recommendation | Priority | Implementation Notes |
|---|---|---|---|
| 1 | **Replace raw JSON delta with structured diff cards** | HIGH | Use `AuditDeltaEntry[]` from the API. Render each changed field as a labeled row: `Price: $19.99 → $24.99`. Currency fields auto-format with `$`. Long text fields truncate with "Show more". |
| 2 | **Add date-range filter + server-side total count** | HIGH | Add `dateFrom` / `dateTo` to `useQuery`. Display `Showing 1–20 of 142 results`. |
| 3 | **Row-click → Detail Drawer** | MEDIUM | Clicking a row opens a slide-over drawer with full metadata: actor avatar/name/role, IP, user-agent, request-id, resource label, and formatted delta. |
| 4 | **Auto-refresh toggle** | LOW | Add a "Live" toggle that polls `GET /admin/audit` every 15s and prepends new rows with a highlight animation. |
| 5 | **Fix double-pagination bug** | HIGH | Remove server `page` state or align DataTable `pageSize` with fetch `limit=20`. Use server-side pagination exclusively (`page` + `limit` query params). |

### 3.2 Notification Bell — From Dropdown to Inbox

**Current State:**
- `NotificationBell.tsx` fetches list only when opened.
- List does not refresh while open.
- No filters, no deep-linking, no "mark all read" persistence.

**Recommendations:**

| # | Recommendation | Priority | Implementation Notes |
|---|---|---|---|
| 6 | **Add full Notifications Inbox route** (`/notifications/inbox`) | HIGH | Move beyond the dropdown. Create a dedicated page with tabs: All / Unread. Group by date (Today, Yesterday, This Week). |
| 7 | **Deep-linking** | HIGH | Each notification carries `deepLink` or `data.resourceId`. Clicking navigates to the entity (e.g., `/orders/123`). Bell list should render clickable rows, not just "mark read". |
| 8 | **Refresh list while open** | MEDIUM | When the dropdown is open, re-fetch on SSE toast arrival. Use a shared `useNotificationsList()` query that invalidates on `notification.received`. |
| 9 | **Unify severity dot + unread indicator** | LOW | Replace two overlapping dots with a single left-border color indicator (critical=red, warning=amber, info=blue). Unread = filled; read = outline. |
| 10 | **Add filter chips** | LOW | Filter by severity (Critical, Warning, Info) and type (Order, Inventory, Staff). |

### 3.3 Real-Time Architecture — Consolidate Streams

**Current State:**
- Two independent SSE connections to `/api/v1/admin/stream`:
  1. `useAdminRealtime.ts` (toasts)
  2. `openAdminEventStream` (access events → cache invalidation / forced logout)
- Plus a 30s REST poll for unread count.

**Recommendation:**

| # | Recommendation | Priority | Implementation Notes |
|---|---|---|---|
| 11 | **Single SSE connection, fan-out to concerns** | HIGH | Create a `RealtimeProvider` context that opens one `EventSource`. Dispatch events to registered listeners:
  - Toast stack (notifications)
  - Access control handler (session revoked → logout)
  - Unread count invalidator
  - Audit auto-refresh hook
  Reduces connection overhead by ~66% and eliminates race conditions between streams. |

### 3.4 Broadcast & Template Management

| # | Recommendation | Priority | Implementation Notes |
|---|---|---|---|
| 12 | **Broadcast confirmation modal + recipient preview** | MEDIUM | Before sending, show "This will send to X active staff members." Require explicit confirmation. |
| 13 | **Template live preview** | MEDIUM | Add a "Preview" panel that renders the template with sample data (e.g., `{{orderNumber}}` → `TEST-12345`). |
| 14 | **Variable autocomplete in template editor** | LOW | As the admin types `{{`, show a dropdown of available variables (`orderNumber`, `userName`, `trackingNumber`). |

### 3.5 Sessions Page

| # | Recommendation | Priority | Implementation Notes |
|---|---|---|---|
| 15 | **Replace `confirm()` with modal** | HIGH | `Sessions.tsx` uses browser-native `confirm()` for revoke actions. Replace with a branded modal showing device info, location, and "Revoke" / "Cancel" buttons. |
| 16 | **Add device/browser fingerprint** | MEDIUM | Parse `userAgent` into icon + text ("Chrome on Windows", "Safari on iPhone"). Show last active time. |
