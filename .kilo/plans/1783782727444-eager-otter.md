# Nova_Store OMS — Gap Analysis: Rider Management, Order Lifecycle & Business Rules

**Date:** 2026-07-11
**Scope:** Frontend and Backend architectures
**Baseline Functional Requirements:**
1. Rider Management Logic (enrollment, profile storage without dedicated dashboard/login, single/batch order assignment)
2. Order Lifecycle & State Management (strict end-to-end state machine from Pending to Order Complete)
3. Configuration & Business Rules (global or store-level "Pay on Delivery" toggle)

---

## Executive Summary

The Nova_Store OMS has a mature order lifecycle with server-side state guards and comprehensive return workflows, but it lacks structured rider management and configurable payment-method business rules. Delivery personnel are represented as free-text driver names only. The frontend order filters are missing newer dispatch statuses, and the dispatch action relies on a browser `prompt` rather than a managed rider roster. No Pay-on-Delivery (POD) toggle exists anywhere in the backend or frontend settings. The existing RBAC layer (STORE_OWNER / MANAGER / ORDER_STAFF / inventory staff) is robust and can be extended, but new rider and payment endpoints must be explicitly gated to prevent privilege escalation.

## 1. Frontend Architecture

### 1.1 Completed Features
- **Order list with status badges** (`Frontend/src/admin/features/orders/OrdersList.tsx`) renders admin orders with `status` and `delivery_status` visual indicators.
- **Order detail with milestone actions** (`Frontend/src/admin/features/orders/OrderDetail.tsx`) exposes dispatch-ready transitions: "Mark Ready for Dispatch" and "Dispatch Order".
- **Dispatch queue Kanban** (`Frontend/src/admin/features/orders/DispatchQueue.tsx`) surfaces `ready_for_dispatch` and actively en-route orders.
- **Return processing UI** (`Frontend/src/admin/features/orders/Returns.tsx`) supports review, approval, and rejection flows.
- **Role-based UI guards** (`Frontend/src/admin/components/guards/RequirePermission.tsx`, `lib/permissions.ts`) enforce STORE_OWNER/MANAGER/ORDER_STAFF visibility across route maps.
- **Typed API client** (`Frontend/src/admin/features/orders/api/orders.ts`) standardizes fetch/update payloads for order milestones.

### 1.2 Pending / Incomplete Features
- **Rider management UI — ABSENT.** No rider roster, enrollment form, profile view, assignment screen, or batch-assignment interface exists.
- **Incomplete order status filter coverage.** `OrdersList.tsx` only queries for `pending`, `processing`, `completed`, and `cancelled`; it omits `ready_for_dispatch`, `dispatched`, `out_for_delivery`, `delivery_attempted`, and delivery-specific sub-states.
- **Unmanaged driver entry.** `OrderDetail.tsx` dispatches orders via `window.prompt` for free-text driver name. There is no dropdown, search, or roster picker.
- **No Pay-on-Delivery toggle in settings.** `StoreSettings.tsx` exposes `low_stock_threshold` and `allow_backorders` but does not surface payment-method config, nor is there a dedicated Payments settings page.
- **No "Order Complete" terminal-state UX.** The lifecycle ends at `delivered`, `cancelled`, `refunded`, or `returned`, but the UI does not present a unified "Order Complete" milestone or commingle terminal states into a single filter.

### 1.3 Recommended Enhancements
- **Rider Enrollment & Profile UI:** Modal/form to capture first name, last name, phone, address, ID/vehicle details, and active status. Store personal info client-side only as needed; keep rendering minimal since no rider dashboard is required.
- **Roster picker + batch assignment:** Iterate `DispatchQueue.tsx` or `OrdersList.tsx` to enable selecting multiple orders and assigning them to a rider from a dropdown populated via `/api/admin/riders`.
- **POD toggle in Store Settings:** Add a boolean switch under a new "Payment & Fulfillment" section in `StoreSettings.tsx` posting to `/admin/store/settings`.
- **Unified "Completed" order filter:** Surface a synthetic `completed` filter in `OrdersList.tsx` that maps to `delivered OR returned` to give operators a single lifecycle end-state view.
- **Real-time dispatch board enrichment:** Bind `DispatchQueue.tsx` to rider assignment so en-route columns show rider name and contact, not just driver free-text.

## 2. Backend Architecture

### 2.1 Completed Features
- **Formal order state machine** (`Backend/src/services/order.service.js`) enforces allowed transitions from `pending` → `confirmed` → `processing` → `ready_for_dispatch` → `dispatched` → `out_for_delivery` → `delivery_attempted` → `delivered`, with terminal states `cancelled`, `returned`, `refunded`.
- **Delivery sub-state machine** (`delivery_status` enum on `orders` + `delivery_dispatches` table) tracks assignment, pickup, out-for-delivery, attempts, and returns-to-store.
- **Order status history audit** (`order_status_history` table, `order-status-history.model.js`) logs every transition with `changed_by`.
- **Return lifecycle** (`return_status` enum + `RETURN_TRANSITIONS` map) with MANAGER-only `processReturn` and a 7-day hard customer window.
- **Granular RBAC middleware** (`require-admin.middleware.js`) supports STORE_OWNER, MANAGER, ORDER_STAFF, and INVENTORY_STAFF, with wildcard `*` for owners and computed effective permissions per request.
- **Stores & store scoping** (`053_create_stores_table.sql`, `store.model.js`) introduces `store_id` across tables and a `store_settings` JSONB store for key/value configuration.
- **Settings framework** (`setting.model.js`, `019_settings_seed.sql`) supports grouped global key/value settings with history and typed parsing.

### 2.2 Pending / Incomplete Features
- **Rider entity — MISSING.** No `riders` table, model, service, or API. `delivery_dispatches` stores `driver_name` (TEXT, NOT NULL) and `driver_phone` as free text; there is no FK to a managed roster.
- **Rider assignment APIs — MISSING.** No endpoints for CRUD on riders, no assignment to single orders, and no batch-assignment endpoint.
- **Pay-on-Delivery configuration — MISSING.** No `payment.pay_on_delivery_enabled` (or equivalent) key in `settings` or `store_settings`. No COD/POD validation in checkout or dispatch flows.
- **"Order Complete" terminal state — ABSENT.** The machine uses `delivered` as fulfillment terminal, but there is no consolidated `completed` state, nor is the relationship between `delivered` + return window expiry and "complete" enforced server-side.
- **Setting exposure gap.** `store_settings` supports arbitrary key/value pairs, but the frontend currently exposes only `low_stock_threshold` and `allow_backorders`. Payment-method toggles are not surfaced.

### 2.3 Recommended Enhancements
- **Rider model & persistence:** Add `riders` table (`id`, `first_name`, `last_name`, `phone`, `address_jsonb`, `id_type`, `id_number`, `vehicle_type`, `vehicle_registration`, `is_active`, `store_id`, `created_by`, timestamps). Sequelize model with scoped active/inactive queries.
- **Rider management APIs:** CRUD under `/admin/riders` gated to `rider:read` / `rider:write`. Prevent `is_active=false` riders from being assigned.
- **Assignment APIs:** Enhance `/orders/admin/:id/dispatch` (or a dedicated `/assign-rider`) to accept `rider_id` instead of free-text name. Add `/orders/admin/bulk-assign-rider` for batch operations.
- **POD toggle:** Seed `payment.pay_on_delivery_enabled` (boolean) in `store_settings` or `settings`. Add server-side guard in checkout and dispatch flows to reject POD selection when disabled. Expose in `PATCH /admin/store/settings`.
- **Unified order-complete transition:** Introduce a `completeOrder` transition that sets `status = 'completed'`, closes the return window, and prevents further status changes, while preserving `delivered_at` and `delivery_status`.

## 4. Security & Permission Gap Summary

| New Capability | Proposed Permission Slug | Allowed Roles | Backend Middleware | Frontend Guard |
|----------------|------------------------|---------------|-------------------|----------------|
| Rider CRUD | `rider:read`, `rider:write` | STORE_OWNER, MANAGER (write); STORE_OWNER, MANAGER, ORDER_STAFF (read) | `requireAnyPermission('rider:read','rider:write')` or `requireManager` for mutations | `RequirePermission slug="rider:read"` |
| Rider Assignment (single) | `rider:assign` | STORE_OWNER, MANAGER, ORDER_STAFF | `requireOrderStaff` (existing) | Existing order detail guard |
| Batch Assignment | `rider:batch_assign` | STORE_OWNER, MANAGER | `requireManager` | Existing bulk-action guards |
| POD Toggle | `payment:write` | STORE_OWNER, MANAGER | `requireManager` | `RequirePermission slug="payment:write"` |
| View POD Status | `payment:read` | STORE_OWNER, MANAGER, ORDER_STAFF | `protect` + `hasAnyPermission('payment:read','*')` | Route guard |

**Critical RBAC note:** The wildcard `*` permission on STORE_OWNER bypasses all slug checks. New rider endpoints must **not** rely on deny-list overrides for owners; instead, they should gate mutations to MANAGER+ and expose read-only data to ORDER_STAFF without personal-ID fields. Use `hasAnyPermission` to ensure non-owner managers still face explicit ACLs.

## 5. Migration & Data Considerations

- **Rider backfill:** Existing `delivery_dispatches.driver_name` records will need a backfill placeholder or be linked to a manually created "Unassigned Rider" record if strict FK integrity is enforced.
- **Single-store hardcode:** `sql/060_single_store_hardcode.sql` indicates `SINGLE_STORE_ID` is still used; fleet/riders must respect this until multi-store is fully lifted.
- **Store settings schema:** `store_settings` is JSONB and already stores `low_stock_threshold` and `allow_backorders`. New keys (`payment.pay_on_delivery_enabled`) fit without migration.
- **Order status enum migration:** Adding `completed` requires a constraint update (`047_manual_delivery_and_returns.sql`) and a data migration for existing `delivered` orders if consolidation is desired.
