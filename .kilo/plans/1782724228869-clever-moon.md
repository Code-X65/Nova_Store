# Plan: Fill Critical Validation & Requirement Gaps for Products, Categories, and Brands

## Date: 2026-06-29
## Author: Kilo (Planner)

---

## 1. Problem Statement

The user observed that **color** is not requested when creating a standard product. After auditing the codebase, several critical fields exist in the database schema but are **missing from API validation**, and there are validation gaps across Brands and Categories.

---

## 2. Current State Analysis

### 2.1 Product Creation (`product.routes.js` + `productSchema`)
**Required by Joi validator:** `sku`, `name`, `category_id`, `price`

**DB columns that exist but are NOT in the Joi schema:**
| DB Column | Purpose | Critical? |
|-----------|---------|-----------|
| `color` | Primary product color | **High** — User's exact concern |
| `weight` | Shipping calculations | **High** |
| `dimensions` | Shipping/box sizing | **High** |
| `cost_price` | Profit margin analytics | **Medium** |
| `allow_backorder` | Inventory behavior | **Medium** |
| `track_inventory` | Inventory behavior | **Medium** |
| `tags` | Search/discovery | **Medium** |
| `meta_title` | SEO | **Low** |
| `meta_description` | SEO | **Low** |
| `meta_keywords` | SEO | **Low** |

**Color is only accessible today via:**
- `product_variants.option_values` (e.g., `{"color": "Red"}`)
- Dynamic `category_attributes` (category-defined)

There is **no first-class `color` column** on the `products` table, but adding one is reasonable.

### 2.2 Category Creation (`category.validator.js`)
**Required by Joi validator:** `name`, `description`, `image_url`, `thumbnail_url`, `icon`

**Problem:** The DB schema marks `description`, `image_url`, `thumbnail_url` as nullable (`TEXT`), but the API requires them. This is overly strict.

**Missing from category schema:**
- `color` — Useful for UI badges/filters
- `brand_id` — Some taxonomies link categories to brands (e.g., "Apple Watches" under Apple)

### 2.3 Brand Creation (`product-brand.routes.js`)
**Problem:** There is **no `brand.validator.js` file** and no validation middleware applied to brand routes.

**Current behavior:**
- Only `name` is implicitly required (DB constraint)
- No duplicate slug/name checking
- No input sanitization beyond what the DB does

**Missing from brand schema/validation:**
- `logo_url` validation
- `website_url` validation
- `thumbnail_url` validation
- Slug uniqueness check

---

## 3. Proposed Changes

### 3.1 Product Schema & Routes
1. **Add `color` field** to product creation/update schema (string, optional, default `null`)
2. **Add `weight` and `dimensions`** to schema (optional — critical for shipping)
3. **Add `cost_price`** to schema (optional — for analytics)
4. **Add `tags`** to schema (array of strings, optional)
5. **Add `allow_backorder` and `track_inventory`** to schema (boolean, optional)
6. **Add `meta_title`, `meta_description`, `meta_keywords`** to schema (optional)
7. **Add `color` to Swagger documentation**

### 3.2 Category Schema & Routes
1. **Relax `category.validator.js`** — make `description`, `image_url`, `thumbnail_url` optional (nullable in DB)
2. **Add `color`** to category schema (optional string)
3. **Add `brand_id`** to category schema (optional UUID, validated against existing brands)
4. **Add `color` and `brand_id` to Swagger documentation**

### 3.3 Brand Schema & Routes
1. **Create `brand.validator.js`** with proper Joi validation
2. **Add validation middleware** to all brand routes
3. **Add duplicate slug/name checking** in `brand.controller.js` (similar to category)
4. **Add Swagger documentation for validations**

---

## 4. Database Changes Needed

### 4.1 Add `color` to products table
```sql
ALTER TABLE public.products
  ADD COLUMN IF NOT EXISTS color TEXT;
```

### 4.2 Add `color` to product_categories table
```sql
ALTER TABLE public.product_categories
  ADD COLUMN IF NOT EXISTS color TEXT;
```

---

## 6. Risks & Tradeoffs

- **Adding `color` to products** duplicates what variants already do. However, it provides a simple filter for simple products without variants. Proposal: keep both approaches.
- **Relaxing category validation** could break existing callers that assume those fields exist. Proposal: make them optional in the validator but keep them in the response.
- **Brand validation** currently has no middleware; adding it may break existing API consumers that send extra fields. Proposal: use `.unknown(false)` in Joi to reject extra fields, which is safer long-term.
