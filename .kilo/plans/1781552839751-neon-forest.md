# Appendix A: SuperAdmin + Admin RBAC with Invitations (Backend Plan)

> **Scope**: Backend only  
> **Goal**: Replace the current dual/single admin model with a proper SuperAdmin + Admin RBAC system powered by email invitations.

---

## A.1 Current State Diagnosis

### Dual Admin Systems (Critical Bug)

The codebase runs **two parallel admin systems** that do not interoperate:

| System | Table | Used By | Auth Method |
|--------|-------|---------|-------------|
| A | `admins` | `require-admin.middleware.js` → `admin.auth.controller.js` | Session cookie (`req.session.adminId`) |
| B | `users` (role=`ADMIN`) | `auth.service.js` → `auth.controller.js` | JWT Bearer token |

**Consequences:**
- An admin created via System B cannot log in via System A's session middleware.
- `adminRegister()` exists in `auth.service.js` but is **blocked with 403** in `auth.controller.js:491-498`.
- `adminLogin()` in `auth.service.js:247` only allows the single `ADMIN_EMAIL` env var — multi-admin login is impossible.
- `require-admin.middleware.js` loads permissions from the `users` table by email match, falling back to `['*']` (wildcard super-admin) if no match found — **security risk**.

### No SUPER_ADMIN Role
- `sql/005_seed_default_roles_permissions.sql` seeds `admin`, `moderator`, `customer` — **no `SUPER_ADMIN`**.
- Code references `SUPER_ADMIN` in `auth.service.js:127` and `user.model.js:231` but the role does not exist in the database.

### No Invitation System
- Zero invitation-related code exists. No `invitations` table, no token generation, no email flow, no acceptance endpoint.

---

## A.2 Target Architecture

### Design Decision: Consolidate to `users` Table

**Decision**: Deprecate the `admins` table. All admin/superadmin identities live in the `users` table, differentiated by RBAC roles via `user_roles`.

**Rationale:**
- Single source of truth — eliminates dual-login confusion.
- `users` table already supports: OAuth, lockout, password reset, email verification, avatar, audit trail.
- `user_roles` + `role_permissions` already provide the RBAC foundation.
- `admins` table becomes a backwards-compatibility shim during migration only.

### Role Model

```
SUPER_ADMIN  →  Full system access. Can invite admins, assign permissions, manage all admins.
ADMIN        →  Scoped access. Permissions assigned by SUPER_ADMIN at invite time.
MODERATOR    →  (Existing) Content moderation.
CUSTOMER     →  (Existing) Shopper.
```

## A.4 Detailed Implementation Steps

### Phase 1: Database Foundation (Migration 045)

**File**: `Backend/sql/045_superadmin_rbac_invitations.sql`

```sql
-- 1. Create invitations table
CREATE TABLE IF NOT EXISTS invitations (
  id UUID DEFAULT uuid_generate_v4() PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  token CHAR(64) NOT NULL UNIQUE,
  role_id UUID REFERENCES roles(id) ON DELETE RESTRICT,
  permissions JSONB DEFAULT '[]'::jsonb,
  invited_by UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
  accepted_at TIMESTAMP WITH TIME ZONE,
  status VARCHAR(20) DEFAULT 'pending' CHECK (status IN ('pending','accepted','expired','revoked')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_invitations_token ON invitations(token);
CREATE INDEX IF NOT EXISTS idx_invitations_email ON invitations(email);
CREATE INDEX IF NOT EXISTS idx_invitations_status ON invitations(status);
CREATE INDEX IF NOT EXISTS idx_invitations_invited_by ON invitations(invited_by);

ALTER TABLE invitations DISABLE ROW LEVEL SECURITY;

-- 2. Add SUPER_ADMIN role (if not already seeded, idempotent)
INSERT INTO roles (id, name, description, created_at, updated_at)
VALUES (
  '00000000-0000-0000-0000-000000000001',
  'SUPER_ADMIN',
  'Full system access — can invite and manage all admins',
  NOW(), NOW()
)
ON CONFLICT (name) DO NOTHING;

-- 3. Grant SUPER_ADMIN all permissions (wildcard)
INSERT INTO role_permissions (role_id, permission_id, created_at)
SELECT '00000000-0000-0000-0000-000000000001', p.id, NOW()
FROM permissions p
ON CONFLICT DO NOTHING;

-- Also grant explicit wildcard if permissions table has '*'
INSERT INTO role_permissions (role_id, permission_id, created_at)
SELECT '00000000-0000-0000-0000-000000000001', p.id, NOW()
FROM permissions p
WHERE p.name = '*'
ON CONFLICT DO NOTHING;

-- 4. Mark admins table as deprecated (add comment, keep table for migration grace period)
COMMENT ON TABLE admins IS 'DEPRECATED — migrated to users + user_roles. Will be dropped after migration verification.';
```

**Seed script** (`Backend/src/scripts/seed-superadmin.js` or inline in run-migrations):

- If not, find the current admin (the one seeded by `025_seed_admin_user.sql`).
- Assign `SUPER_ADMIN` role to that user.
- Log: "Promoted existing admin to SUPER_ADMIN".

## A.8 Security Considerations

| Risk | Mitigation |
|------|-----------|
| Invitation token brute-force | 64-char hex token = 2^256 possibilities. Rate-limit accept-invite endpoint. |
| Token reuse after acceptance | Token status checked — accepted/expired/revoked tokens rejected. |
| Unauthorized admin creation | `createInvitation` checks inviter is SUPER_ADMIN. No self-registration for admins. |
| Permission escalation | SUPER_ADMIN wildcard `['*']` is hardcoded in role_permissions, not in middleware. Cannot be bypassed. |
| Invitation email enumeration | Accept-invite endpoint returns generic "invalid or expired token" for all failure cases. |
| Stale invitations | Cleanup job runs daily to expire/clean old invitations. |

## A.11 Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Breaking existing admin sessions | HIGH | Deploy middleware update during low-traffic window. Add fallback to old `admins` table for 1 release. |
| SUPER_ADMIN role already exists in some env | LOW | Use `ON CONFLICT DO NOTHING` in seed SQL. |
| Invitation email spam | MEDIUM | Rate-limit invite creation (max 10/hour per SUPER_ADMIN). |
| Orphaned invitations | LOW | Cleanup job + manual revoke endpoint. |
| Permission misconfiguration | HIGH | SuperAdmin UI shows current permissions before saving. Audit log all permission changes. |
